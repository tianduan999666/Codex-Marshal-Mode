# V4-Target 蓝图

最后更新：2026-03-29
状态：现行标准件
适用阶段：`V4-Target` 准备阶段
来源：当前仓 `V4-Trial` 验收结果 + 旧参考仓 Target 蓝图提炼

## 一句话结论

`V4-Target` 不是重起一套新东西，而是把当前仓已验证可行的 Trial 能力，收口成一个更稳定、可迁移、可持续维护的自治研发底座。

## 北极星目标

- 对主公：只需定目标、定边界、给资源、拍关键板。
- 对系统：负责承接任务、维护状态、积累证据、支持恢复、保证收口。
- 对维护者：能解释、能迁移、能逐步增强，不因补丁式演进再次散掉。

## 当前现实基础

当前仓已经完成：

1. `V4-Trial` 三条真实试跑闭环
2. 任务包 5 件套标准化
3. 面板入口验收口径
4. 执行区现行件 / 证据稿边界
5. 跨轮恢复说明与样本
6. 本地安全提交流程
7. 任务包半自动起包脚手架
8. 执行区旧证据稿第一轮归档

## Target 阶段的目标状态

- 入口保持官方面板优先，但口径更稳、恢复更强、维护动作更少。
- 任务包继续作为唯一运行态目标，且支持更稳的起包、恢复、收口。
- 现行标准件继续集中在当前目录内，避免外部路径依赖回潮。
- 维护动作有更稳定的流程层支撑，而不是继续靠人工记忆。

## 设计原则

1. 先稳态，再扩功能。
2. 先维持当前目录内自含，再谈更远的迁移包装。
3. 先让维护层动作可解释、可复用，再考虑更强自动化。
4. 先保证 Trial 验证出的核心能力不回退，再扩下一层能力。

## Target 核心模块

### 1. 入口层

- 继续以官方面板为默认入口。
- 保持丞相模式触发、状态汇报与人工验收闭环稳定。

### 2. 控制平面层

- 任务包 5 件套继续作为最小控制平面。
- 激活任务、决策、拍板、结果继续文件化。

### 3. 维护层

- 维护层动作保留在当前目录。
- 已形成的安全提交、半自动起包、归档规则继续可用。

### 4. 证据与归档层

- 执行区保持现行件少而稳。
- 证据稿进入归档区，减少现行区噪音。

## 当前硬边界

- 不重开新仓，不改已锁定目录结构。
- 不整体迁入旧仓 `codex-home-export/`。
- 不立即扩展独立前端、多 Agent、长期记忆系统。
- 不在当前轮直接进入重量级安装链重构。

## 进入 Target 的最小条件

- `V4-Trial` 已通过验收。
- 三条真实试跑闭环已形成可追溯证据。
- 稳定性、起包、归档三项后置治理已有现行方案。
- 当前目录可以作为下一阶段唯一准备落点。

## 当前不直接做的事

- 不直接开始大规模功能开发。
- 不直接宣布独立安装链已成立。
- 不直接把 Trial 文件体系推翻重做。

## 本稿结论

`V4-Target` 的正确打开方式，不是“推翻重来”，而是“在当前仓已验证能力之上，做稳态升级与工程化收口”。
