# 短期困难长期收益调整建议

最后更新：2026-04-03
状态：待评审
优先级：中
决策原则：短期可能困难，长期更大收益

## 一、调整原则

基于 VibeCoding 方案 B 的成功实施经验，识别其他文档中可以应用"短期困难，长期收益"原则的部分。

**核心判断标准**：
1. 短期实施成本 > 0（需要额外工作）
2. 长期收益显著（质量提升、效率提升、架构优化）
3. 符合"反屎山总纲"理念
4. 为未来扩展铺平道路

## 二、Phase 2-3 风险预警调整建议

### 当前状态分析

**文档**：[11-Phase2-3风险预警与技术准备.md](11-Phase2-3风险预警与技术准备.md)

**当前定位**：风险预警，标注为"必须在 Phase 2 启动前解决"

**问题识别**：
1. 文档定位为"风险预警"，但实际上是"技术准备清单"
2. 降级方案（串行执行）被标记为"保底方案"，暗示可以不做技术准备
3. 缺少"立即启动"的决策依据

### 调整建议：从"风险预警"升级为"Phase 2 前置开发"

**方案 A：保守路线（当前）**
- 定位：风险预警
- 实施时机：Phase 2 启动前
- 降级方案：回退到串行执行
- 短期成本：0（可以不做）
- 长期风险：Phase 2 启动时发现不可用，被迫降级

**方案 B：激进路线（推荐）**
- 定位：Phase 2 前置开发（必做项）
- 实施时机：立即启动（与 Phase 1 并行）
- 降级方案：无（必须完成）
- 短期成本：13-16 小时（2 个工作日）
- 长期收益：
  - Phase 2 并行 Agent 编排可用
  - 文件并发写入安全
  - API 限流自动处理
  - 为 Phase 3 多 Agent 大系统铺平道路

**调整内容**：
1. 文档标题改为"Phase 2 前置开发清单"
2. 状态从"风险预警"改为"前置开发（必做）"
3. 删除降级方案章节（不给自己留退路）
4. 增加"立即启动"的决策依据
5. 将技术准备清单改为"开发任务包"

**决策依据**：
- 文件并发锁和 API 限流是 Phase 2 的基础设施
- 如果不提前开发，Phase 2 启动时会被迫降级到串行执行
- 串行执行会导致 Phase 2 的效率提升从 1.6 倍降至 1.2 倍
- 短期 2 天的投入，换取长期的架构稳定性

## 三、规划策略候选规范调整建议

### 当前状态分析

**文档**：[07-V4-规划策略候选规范.md](07-V4-规划策略候选规范.md)

**当前定位**：候选规范（可选）

**问题识别**：
1. 标题为"候选规范"，暗示可以不采纳
2. 规划策略是复杂任务的必需品，不是可选项
3. 与 VibeCoding 方案 B 的"规划-执行强制分离"理念一致

### 调整建议：从"候选规范"升级为"现行规范"

**方案 A：保守路线（当前）**
- 定位：候选规范（可选）
- 适用范围：高不确定性任务
- 强制性：无
- 短期成本：0（可以不用）
- 长期风险：复杂任务缺少规划，返工率高

**方案 B：激进路线（推荐）**
- 定位：现行规范（必用）
- 适用范围：所有 estimated_hours > 4 的任务
- 强制性：门禁检查
- 短期成本：1-2 小时（修改门禁脚本）
- 长期收益：
  - 与 tech-spec.md 形成完整的规划体系
  - 主假设、最小推进步、验证信号成为标配
  - 路线修正有据可依
  - 为 Phase 2 Planner Agent 提供输入格式

**调整内容**：
1. 文档标题改为"V4-规划策略规范"（去掉"候选"）
2. 状态从"候选规范"改为"现行规范"
3. 在 contract.yaml 中增加必填字段：planning_hint（复杂任务）
4. 在 state.yaml 中增加必填字段：plan_step, verify_signal（复杂任务）
5. 门禁脚本检查：复杂任务必须填写规划字段

**决策依据**：
- VibeCoding 方案 B 已经证明"规划-执行强制分离"的价值
- 规划策略是 tech-spec.md 的补充，两者形成完整体系
- 短期 1-2 小时的投入，换取长期的任务质量提升
