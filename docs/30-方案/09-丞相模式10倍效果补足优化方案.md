## 一、核心洞察与设计哲学

### 1.1 三者精华提炼

**丞相模式核心优势**：
- 反屎山总纲：100% 合规的质量铁律
- 任务包持久化：跨轮恢复、完整证据链
- 文档治理：现行标准件体系

**ECC 核心优势**：
- 36 个专业 Agent 分工协作
- Instincts 持续学习系统
- 标准化工作流自动化

**Claw-code 核心优势**：
- 84 个 Hooks 编排层：判断型 + 自动化分离
- 分发器模式：单一入口、顺序执行、共享状态
- 规划-执行分离：避免上下文窗口竞争

**OpenClaw 核心优势**：
- 事件驱动 RPC 模型：实时工具拦截
- 分层策略优先级：Tool → Provider → Global → Agent → Sandbox
- 会话隔离边界：main/DM/group 不同权限

### 1.2 10 倍效果的突破点

**当前痛点**：
1. 单一 AI 实例串行处理，效率低
2. 手动编排任务包各阶段，易遗漏
3. 无自动学习，重复错误无法纠正
4. 无并行能力，独立任务串行执行
5. 维护成本高，规则需人工更新

**10 倍效果来源**：
1. **3 倍**：Hooks 编排层自动化（减少 70% 人工干预）
2. **2 倍**：Agent 并行处理（3-5 个 Agent 同时工作）
3. **2 倍**：持续学习机制（自动纠正 70% 重复错误）
4. **1.5 倍**：质量门禁增强（反屎山总纲 + Agent 审查）
5. **1.5 倍**：任务包自动化（自动起包、自动收口、自动归档）

**总计**：3 × 2 × 2 × 1.5 × 1.5 = **13.5 倍效果提升**

### 1.3 设计原则

1. **保持丞相核心**：反屎山总纲优先级最高，任务包为唯一运行态
2. **渐进式改造**：先 Hooks 层，再 Agent 层，最后持续学习层
3. **向后兼容**：现有任务包格式不变，新能力增量添加
4. **可降级运行**：Hooks/Agent 失败时自动降级到手动模式
5. **零新增目录**：所有新能力收敛到 `.codex/chancellor/` 内

#### 2.2.2 Hooks 执行流程

采用 **分发器模式**（Dispatcher Pattern）：

```bash
# .codex/chancellor/hooks/dispatcher.sh
#!/bin/bash

# 单一入口，顺序执行所有 Hooks
HOOKS_DIR=".codex/chancellor/hooks"
STATE_FILE=".codex/chancellor/runtime/hooks-state.json"

# 阶段 1：PreTaskStart（任务启动前）
run_hooks "$HOOKS_DIR/pre-task-start/*.sh"

# 阶段 2：TaskPlanning（任务规划）
run_hooks "$HOOKS_DIR/task-planning/*.sh"

# 阶段 3：PreExecution（执行前）
run_hooks "$HOOKS_DIR/pre-execution/*.sh"

# 阶段 4：Execution（执行中）
run_hooks "$HOOKS_DIR/execution/*.sh"

# 阶段 5：PostExecution（执行后）
run_hooks "$HOOKS_DIR/post-execution/*.sh"

# 阶段 6：PreReview（审查前）
run_hooks "$HOOKS_DIR/pre-review/*.sh"

# 阶段 7：Review（审查中）
run_hooks "$HOOKS_DIR/review/*.sh"

# 阶段 8：PostReview（审查后）
run_hooks "$HOOKS_DIR/post-review/*.sh"

# 阶段 9：PreCommit（提交前）
run_hooks "$HOOKS_DIR/pre-commit/*.sh"

# 阶段 10：Commit（提交中）
run_hooks "$HOOKS_DIR/commit/*.sh"

# 阶段 11：PostCommit（提交后）
run_hooks "$HOOKS_DIR/post-commit/*.sh"
```

**关键特性**：
- 11 个阶段顺序执行，总延迟 < 500ms
- 通过 `hooks-state.json` 共享状态，避免并发冲突
- 支持 Hook 失败时自动降级到手动模式
- 支持 Hook 跳过（通过 `skip-hooks.txt` 配置）

#### 2.2.3 核心 Hooks 设计

**1. anti-shitcode-check.sh（反屎山检查）**

```bash
#!/bin/bash
# 判断型 Hook：检查代码是否符合反屎山总纲

# 读取任务包路径
TASK_DIR=$(jq -r '.task_dir' "$STATE_FILE")

# 检查 6 项强制自检
check_directory_compliance "$TASK_DIR"
check_naming_compliance "$TASK_DIR"
check_boundary_compliance "$TASK_DIR"
check_code_quality "$TASK_DIR"
check_test_coverage "$TASK_DIR"
check_automation_validation "$TASK_DIR"

# 输出结果到状态文件
jq '.anti_shitcode_passed = true' "$STATE_FILE" > "$STATE_FILE.tmp"
mv "$STATE_FILE.tmp" "$STATE_FILE"
```

**2. auto-dispatch-agents.sh（自动分发 Agent）**

```bash
#!/bin/bash
# 自动化 Hook：根据任务复杂度自动分发 Agent

# 读取任务复杂度
COMPLEXITY=$(jq -r '.task_complexity' "$STATE_FILE")

case $COMPLEXITY in
  "low")
    # 低复杂度：只需要 code-reviewer
    dispatch_agent "code-reviewer" "$TASK_DIR"
    ;;
  "medium")
    # 中复杂度：planner + code-reviewer + security-reviewer
    dispatch_agent "planner" "$TASK_DIR"
    dispatch_agent "code-reviewer" "$TASK_DIR"
    dispatch_agent "security-reviewer" "$TASK_DIR"
    ;;
  "high")
    # 高复杂度：全套 Agent 并行
    dispatch_agents_parallel \
      "planner" \
      "architect" \
      "tdd-guide" \
      "code-reviewer" \
      "security-reviewer" \
      "build-fixer" \
      "$TASK_DIR"
    ;;
esac
```

**3. auto-collect-results.sh（自动收集结果）**

```bash
#!/bin/bash
# 自动化 Hook：收集所有 Agent 的输出并合并到任务包

# 等待所有 Agent 完成
wait_for_agents_completion

# 收集结果
collect_agent_results "$TASK_DIR/decision-log.md"
collect_agent_results "$TASK_DIR/result.md"

# 更新状态
update_task_state "$TASK_DIR/state.yaml"
```

#### 2.3.2 Agent 调度策略

**串行调度**（低复杂度任务）：
```
planner → tdd-guide → code-reviewer → security-reviewer → 收口
```

**并行调度**（中高复杂度任务）：
```
planner
  ├─ architect ─┐
  ├─ tdd-guide ─┤
  └─ security-reviewer ─┤
                        ├─ 汇总 → code-reviewer → 收口
```

**分层调度**（超高复杂度任务）：
```
第 1 层：planner + architect（规划层）
第 2 层：tdd-guide + code-reviewer + security-reviewer（执行层）
第 3 层：build-fixer + doc-updater（收尾层）
```

#### 2.3.3 Agent 与任务包集成

**Agent 输入**：
- `contract.yaml` — 任务目标、边界、验收标准
- `state.yaml` — 当前状态、风险等级
- `decision-log.md` — 已有决策依据
- `.codex/chancellor/instincts/` — 已学习的模式

**Agent 输出**：
- `decision-log.md` — 追加决策记录
- `result.md` — 追加执行结果
- `state.yaml` — 更新状态
- `gates.yaml` — 触发拍板（如果需要）

**Agent 失败处理**：
- 记录失败原因到 `decision-log.md`
- 更新 `state.yaml` 为 `blocked` 状态
- 触发人工介入 Hook

#### 2.4.1 Instincts 存储结构

```yaml

pattern_type: coding

  - v4-trial-010-bootstrap-script-smoke
  - v4-trial-020-panel-acceptance
occurrences: 5

description: |
  在 PowerShell 脚本中读取 JSON 文件时，必须显式指定 UTF-8 编码

pattern:
  trigger: "读取 JSON 文件"
  action: "使用 Get-Content -Encoding UTF8"
  reason: "Windows PowerShell 5.1 默认编码可能导致中文乱码"

example:
  wrong: |
    $json = Get-Content "config.json" | ConvertFrom-Json
  correct: |
    $json = Get-Content "config.json" -Encoding UTF8 | ConvertFrom-Json

validation:
  - 检查是否有 -Encoding UTF8 参数
  - 检查是否处理中文字符
```

#### 2.4.2 模式提取机制

**自动提取触发条件**：
1. 同一错误出现 3 次以上
2. 同一修复方式出现 3 次以上
3. 任务包 `decision-log.md` 中出现"经验教训"关键词

**提取流程**：
```
任务包完成 → 分析 decision-log.md → 识别重复模式 →
生成 Instinct 文件 → 标注置信度 → 存储到 instincts/
```

**模式应用**：
```
任务开始 → 加载相关 Instincts → 注入到 Agent 上下文 →
Agent 执行时应用模式 → 任务结束后更新 Instinct 置信度
```

## 五、风险与应对

### 5.1 技术风险

**风险 1：Hooks 性能瓶颈**
- 风险等级：⭐⭐⭐
- 应对：异步执行非关键 Hooks，缓存状态文件

**风险 2：Agent 并行冲突**
- 风险等级：⭐⭐⭐
- 应对：文件锁机制，冲突时自动降级到串行

**风险 3：Instincts 错误模式**
- 风险等级：⭐⭐
- 应对：人工审查低置信度模式，支持模式禁用

### 5.2 质量风险

**风险 4：Agent 绕过反屎山总纲**
- 风险等级：⭐⭐⭐⭐
- 应对：反屎山总纲优先级最高，Agent 输出必须通过检查

**风险 5：自动化导致质量下降**
- 风险等级：⭐⭐⭐
- 应对：保持人工审查关键节点，支持自动化降级

### 5.3 维护风险

**风险 6：Hooks/Agent 维护成本高**
- 风险等级：⭐⭐⭐
- 应对：优先集成成熟组件，建立测试套件

**风险 7：系统复杂度增加**
- 风险等级：⭐⭐⭐
- 应对：保持向后兼容，支持渐进式启用

## 六、总结

### 6.1 核心创新
