# V4-OS 参考技术采纳评估

最后更新：2026-03-29
状态：现行方案评估件
适用阶段：`V4-Target` 准备阶段
说明：本文件用于评估外部参考技术能否被抽象为 `V4 OS` 的策略、模块或调度逻辑，并明确当前阶段采纳、暂缓或舍弃的边界。

## 一句话结论

当前只采纳“规划策略抽象”与“治理审计原则”两类高复利方向作为后续候选；浏览器执行器、浏览器侧边栏 UI、多代理协议都不进入当前冻结边界；所有 `/v4/...` 新目录提案一律不直接落盘。

## 当前裁决原则

1. 只采纳能力抽象，不绑定具体产品名作为现行模块名。
2. 只在当前目录内留方案文档，不新增未批准目录结构。
3. 只收对 `V4-Target` 长期复利高、且不破坏当前冻结边界的部分。
4. 与当前硬边界冲突的方案，宁可暂缓或舍弃当前提法，也不抢跑实现。

## 必须明确舍弃的部分

- 舍弃把参考项原样落为 `/v4/planners/...`、`/v4/executors/...`、`/v4/ui/...`、`/v4/governance/...`、`/v4/protocols/...` 的目录方案。
- 舍弃把产品名直接写成当前现行模块名的做法。
- 舍弃当前阶段直接启动独立 UI、多 Agent、浏览器自动化主能力的做法。

## 当前允许保留的抽象

### 1. 规划策略候选

- 可保留“任务假设生成 → 多步骤规划 → 验证循环 → 路线修正”的抽象链条。
- 当前已收成候选规范：`docs/30-方案/07-V4-规划策略候选规范.md`
- 现阶段仍不单独落实现模块。

### 2. 治理审计候选

- 可保留“模型路由审计、配置哈希校验、关键响应留痕”的治理思路。
- 当前只作为未来治理层输入，不启动路由系统建设。

## 为什么现在不采浏览器执行与多代理

- 这两类能力都属于高杠杆但高扩面动作。
- 当前仓已冻结“先稳态、再扩功能”的顺序。
- 若现在引入，会同时牵动入口、运行态、验收口径与目录边界，超出本轮最小推进点。

## 对用户参考文档的最终吸收方式

- 吸收“规划策略”这条线，但改写为当前仓内的方案候选，不沿用新目录提案。
- 吸收“治理审计原则”这条线，但只保留原则，不提前上实现。
- 浏览器执行器、浏览器侧边栏、多代理协议，统一转为远期候选，不进入当前现行冻结面。
