# V4 MVP 边界清单

最后更新：2026-03-28
状态：现行标准件
适用阶段：`Phase 1` 至 `V4-Trial` 验收前
来源：当前仓现行口径；历史来源仅作提炼依据，不构成现行运行依赖

## 一句话结论

当前轮 `V4 MVP` 只解决三件事：统一入口语义、任务包运行态、最小验收闭环，并且所有现行产物必须保持当前目录内自含、独立、可迁移。

## 本轮目标

- 保持官方面板为默认入口。
- 固化任务包为唯一运行态目标结构。
- 固化最小验收口径，支撑试跑与恢复。
- 让 `MVP` 与后续完成品都在当前目录内自含落地，便于后续整体迁移。
- 为后续任务包模板和入口验收建立稳定前提。

## 允许迁入的旧仓资产

本轮只迁移旧仓中的 4 类决策资产：

1. 真实需求与范围界定
2. MVP 控制平面
3. 验收清单
4. 任务包目录规范

## 明确不迁移的旧仓内容

- `codex-home-export/` 整包
- `tests/` 保护性测试集合
- 旧 `v4-state/` 运行态目录
- 发布链、第二机链路、便携包链路
- 独立前端、独立命令行、多 Agent、长期记忆

## 本轮必须做

1. 形成唯一现行边界文档。
2. 确认 `.codex/chancellor/tasks/<task-id>/` 为唯一运行态目标。
3. 确认入口负责触发，任务包负责承载。
4. 确认试跑只按最小验收口径推进。
5. 确认现行 `MVP` 文档、规则、模板与运行态文件均收敛在当前目录，不依赖旧仓存活。

## 本轮明确不做

1. 不重构安装链。
2. 不迁整包旧资产。
3. 不恢复测试矩阵、发布链与第二机链路。
4. 不在任务包内承诺新增 `archive/` 子目录。

## 最小验收口径

- 主公无需先判断入口，应直接从面板进入。
- 同一任务能跨轮恢复，不依赖长聊天历史。
- 风险、决策、结果能稳定写入文件。
- 运行态与文档区彻底分离。
- 把当前目录完整拷走后，现行 `MVP` 仍能被理解、继续维护与迁移，不依赖旧仓路径。
