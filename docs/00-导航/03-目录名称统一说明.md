# 目录名称边界说明

最后更新：2026-04-03
状态：现行标准件
适用范围：判断“本地目录名”和“产品命名”是否冲突时使用

## 一句话结论

产品名已经统一为 `丞相模式 / Chancellor Mode`。
本机历史目录名可以继续存在，它不再决定产品口径，也不会影响安装、升级和使用。

## 当前应以什么为准

| 项目 | 现行口径 | 说明 |
| --- | --- | --- |
| 对外中文名 | `丞相模式` | 用户可见统一这样叫 |
| 对外英文名 | `Chancellor Mode` | 文档、工程口径统一如此 |
| GitHub 仓库名 | `Codex-Chancellor-Mode` | 其他电脑安装时直接 clone 这个仓 |
| 本机历史路径 | 允许保留 | 例如当前这台机器上的旧目录名只是本机路径，不代表产品仍沿用旧称呼 |

## 什么时候需要改本机目录名

| 场景 | 是否必须改 | 处理建议 |
| --- | --- | --- |
| 只是本机自己继续开发 | 否 | 直接继续用，不影响功能 |
| 要截图给别人看本地路径 | 否 | 只要 README、仓库名、面板口径一致即可 |
| 想让本机路径也完全统一 | 可选 | 这是整理动作，不是功能前提 |
| 换新电脑重新安装 | 否 | 直接 `git clone` 新仓即可，自然会拿到新目录名 |

## 当前正确理解

```text
产品口径 = 丞相模式 / Chancellor Mode
仓库口径 = Codex-Chancellor-Mode
本机旧路径 = 历史残留，不等于产品仍未改名
```

## 不再建议做的事

- 不再把“是否改本机目录名”当成版本切换前置条件。
- 不再因为历史路径还是旧目录名，就误判系统仍未完成命名统一。
- 不再为此额外安排重命名目录、修快捷方式、补说明文案这一整套动作。

## 结论

判断是否是新版本，看这 3 个点，不看本机旧目录名：

1. GitHub 仓库是不是 `Codex-Chancellor-Mode`
2. `传令：版本` 返回的版本号和真源路径是否正常
3. 面板入口是否只认 `传令：XXXX`
