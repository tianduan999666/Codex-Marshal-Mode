# 拍板结果回写模板

最后更新：2026-03-29
状态：现行维护标准件
适用范围：当前仓维护层拍板完成后的回写动作
说明：本模板用于在主公已拍板后，半自动把结果回写到 `gates.yaml`、`state.yaml`、`decision-log.md`、`result.md`，把任务从 `waiting_gate` 带回可继续推进的状态。

## 一句话结论

当主公已经拍板，且任务需要从 `waiting_gate` 恢复推进时，优先使用当前仓内的拍板结果回写模板，而不是手工分别改四个文件。

## 脚本位置

- `.codex/chancellor/resolve-gate-package.ps1`

## 适用场景

- `gates.yaml` 中已存在 `pending` 状态的待拍板事项。
- 主公已给出明确结论。
- 需要把任务状态从 `waiting_gate` 恢复到 `running`、`ready`、`paused` 或其他真实状态。
- 需要把拍板结果沉淀进任务包运行态。

## 输入项

- `TaskId`
- `GateId`
- `DecisionStatus`（`decided` 或 `dropped`）
- `DecisionSummary`
- `ChosenOption`（可选）
- `NextStatus`（默认 `running`）
- `NextAction`（默认 `按拍板结果继续推进`）

## 输出结果

脚本会同步写入：

- `gates.yaml`：把目标 gate 从 `pending` 回写为 `decided` 或 `dropped`
- `state.yaml`：恢复为新的真实状态，并更新 `next_action`
- `decision-log.md`：追加回写记录
- `result.md`：追加拍板结果摘要
## 使用方式

这是维护层动作，建议在主公已拍板后立刻执行，避免任务长期停在 `waiting_gate`。

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\.codex\chancellor\resolve-gate-package.ps1 `
  -TaskId v4-trial-017-example-task `
  -GateId gate-boundary-change `
  -DecisionStatus decided `
  -DecisionSummary "主公同意按选项 A 推进" `
  -ChosenOption "允许调整" `
  -NextStatus running `
  -NextAction "按选项 A 继续推进目录调整方案"
```

## 边界说明

- 当前脚本只处理单个 `pending` gate 的回写，不自动批量处理多个拍板事项。
- 当前脚本不会自动提交或推送。
- 当前脚本不会替代人工理解拍板内容，只负责把已拍板结果标准化落盘。
- 若存在多个 gate 或异常并存，先按 `docs/40-执行/19-多 gate 与多异常并存处理规则.md` 判断主状态。

## 生成后的动作

1. 回看 `gates.yaml`，确认目标 gate 已更新为 `decided` 或 `dropped`。
2. 回看 `state.yaml`，确认任务已恢复到新的真实状态。
3. 回看 `decision-log.md` 与 `result.md`，确认拍板结论与后续动作已经对齐。
4. 按 `docs/40-执行/14-维护层动作矩阵与收口检查表.md` 做统一收口。

## 长期价值

- 减少拍板完成后的回写遗漏。
- 降低 `waiting_gate` 长期挂起不收口的概率。
- 让拍板闭环从“起手”延伸到“回写完成”。
