# 复杂并存汇报骨架模板

最后更新：2026-03-29
状态：现行维护标准件
适用范围：多 gate 与多异常并存的任务包汇报落盘
说明：本模板用于把复杂并存场景的主状态、主阻塞、次要事项与恢复顺序快速写入任务包，减少复杂裁决后的手工整理成本。

## 一句话结论

当一个任务同时存在多个 gate、多个异常，或 gate 与异常并存时，先按 `19` 选主状态，再用本模板把 `result.md` 与 `decision-log.md` 一次性落盘。

## 什么时候用

- 已按 `docs/40-执行/19-多 gate 与多异常并存处理规则.md` 选出主状态。
- 当前需要把复杂裁决结果同步进任务包。
- 任务已经不适合只靠单一 gate 或单一异常模板表达。

## 最低产出

- `result.md` 写清主状态、主阻塞、主阻塞原因。
- `result.md` 列出次要待处理项、恢复顺序与治理复核。
- `decision-log.md` 记录为何选当前主状态，而不是其他候选状态。
- 提交前按 `docs/30-方案/08-V4-治理审计候选规范.md` 追加治理审计复核。
- 如已确定当前主推进口径，可同步更新 `state.yaml`。

## 最低产出固定槽位

- `主状态落盘`：`result.md` 写清主状态、主阻塞、主阻塞原因。
- `次要事项与恢复顺序`：`result.md` 列出次要待处理项、恢复顺序与治理复核。
- `裁决留痕`：`decision-log.md` 记录为何选当前主状态，而不是其他候选状态。
- `治理复核`：提交前按 `docs/30-方案/08-V4-治理审计候选规范.md` 追加治理审计复核。
- `状态同步`：如已确定当前主推进口径，可同步更新 `state.yaml`。

## 推荐脚本入口

- 脚本：`.codex/chancellor/write-concurrent-status-report.ps1`
- 规则：`docs/40-执行/19-多 gate 与多异常并存处理规则.md`
- 治理：`docs/30-方案/08-V4-治理审计候选规范.md`
- 收口：`docs/40-执行/14-维护层动作矩阵与收口检查表.md`

- 主规则：docs/40-执行/19-多 gate 与多异常并存处理规则.md
- 骨架模板：docs/40-执行/20-复杂并存汇报骨架模板.md

### 次要待处理项

- gate-architecture-choice：待主公拍板
- 回退验证：待环境恢复后补跑

```md
### 恢复顺序

- 先完成环境修复
- 再恢复拍板包准备
- 最后按既定下一步继续推进

### 下一步

- 修复环境并补齐拍板输入

### 治理复核

- 当前主状态依据是否可追溯：待补说明
- 当前次要待处理项是否完整保留：待补说明
- 当前是否发现口径漂移：待复核
- 提交前是否已完成治理审计复核：待确认
```

## `decision-log.md` 推荐骨架

```md
## 2026-03-29 04:20:42

- 决策：记录复杂并存场景主状态为 waiting_assist
- 主阻塞：环境修复未完成
- 原因：当前异常先于拍板阻断推进
- 证据：依据 docs/40-执行/19-多 gate 与多异常并存处理规则.md、docs/40-执行/20-复杂并存汇报骨架模板.md 与 docs/30-方案/08-V4-治理审计候选规范.md 形成统一汇报骨架
- 未选状态：
- waiting_gate：虽有待拍板事项，但当前拍板输入不完整
- paused：当前不是主动停机，而是等待外部协助
```
```md
- 裁决依据：
- 先按下一行动主体判断主状态
- 再按影响范围与阻断顺序确定主阻塞
- 治理提示：复杂裁决结果与提交前，应确认主状态依据、次要待处理项与公开边界已完成治理审计复核
- 影响：result.md 与 decision-log.md 已统一口径
```

## 半自动写入建议

- `TaskId`：目标任务包 ID。
- `PrimaryStatus`：当前唯一主状态。
- `PrimaryBlocker`：当前最先阻断推进的事项。
- `PrimaryReason`：为什么它比其他事项更优先。
- `SecondaryItems`：次要待处理项列表。
- `RecoverySteps`：主阻塞解除后的恢复顺序。
- `NextAction`：本轮最小下一步。
- `DecisionBasis`：本轮裁决依据。
- `RejectedCandidates`：本轮未选状态及原因。
- `SyncState`：如当前主状态已确定，允许同步更新 `state.yaml`。

## 维护层价值

- 把复杂并存场景从口头汇报变成固定骨架。
- 让 `state.yaml`、`result.md`、`decision-log.md` 的口径更稳定。
- 让复杂裁决结果在提交前也能挂上最小治理复核。
- 为后续更强的复杂裁决自动化保留轻量入口。

## 维护层价值固定槽位

- `固定骨架`：把复杂并存场景从口头汇报变成固定骨架。
- `口径稳定`：让 `state.yaml`、`result.md`、`decision-log.md` 的口径更稳定。
- `最小治理复核`：让复杂裁决结果在提交前也能挂上最小治理复核。
- `自动化入口`：为后续更强的复杂裁决自动化保留轻量入口。
