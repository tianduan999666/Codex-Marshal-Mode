# V4 丞相模式封顶边界清单

最后更新：2026-04-05
状态：现行决策件
适用范围：`丞相模式` 本体仓、`codex-home-export/`、本地运行态治理边界
说明：本清单用于冻结当前阶段“必须继续做 / 可以延后做 / 明确不做”的边界，避免丞相模式无限扩张，挤占后续开发闭源业务项目的主时间。

## 一句话结论

当前阶段，丞相模式只封顶为“官方 Codex 面板里的稳定开发控制面”，不以“完整复刻 ECC”或“独立大平台”为目标。

## 三栏总表

| 分类 | 必须继续做 | 可以延后做 | 明确不做 |
| --- | --- | --- | --- |
| 单入口与公开协议 | 继续稳住 `传令：XXXX` 单入口、`状态 / 版本 / 升级` 查询命令、`继续当前任务` 续做链路 | 以后若真实出现高频需求，再补极少量别名或面板快捷说法 | 不重新扩成多命令体系，不搞多套公开触发词并存 |
| 真源与版本纪律 | 继续把开场白、状态栏、边界提示、收口、版本口径锁在 `VERSION.json`；继续执行“受管文件改动必须 bump 版本并重新验真” | 以后可再考虑半自动版本生成或更细粒度版本说明 | 不允许继续出现“同版本不同内容”；不允许多处硬编码公开话术 |
| 安装、升级、自检、回滚 | 继续稳住 `install.cmd / upgrade.cmd / self-check.cmd / rollback.cmd` 这 4 个普通用户维护动作 | 以后可补更友好的维护态摘要输出 | 不把普通用户重新拉回底层 `.ps1` 手工流程；不让安装器静默覆盖用户全局 provider / auth |
| 漂移检查与维护者模式 | 继续保留轻检优先、必要时完整验真、维护时可 `-MaintainerMode` 一次看全漂移 | 以后可把维护态汇总做得更清晰，但仍应按需触发 | 不把“全量漂移清单”设成每次开工必跑默认动作；不让维护输出来淹没日常任务入口 |
| 任务包、收口、留痕 | 继续稳住任务包、收口模板、结果回写、当前真实状态总表 | 以后可从真实任务包里提炼 3-5 个高频复用模板 | 不做大而全的流程模板仓，不做脱离真实任务样本的假复用件 |
| 复用能力沉淀 | 先只提炼真实高频、可验证、可维护的少量复用件 | 后续若样本足够，再做轻量模式沉淀 | 不先做“147 个 skills 覆盖率”这类平台幻觉工程 |
| 多 Agent / Skills / 持续学习 | 当前不作为主线，只记录为远期候选方向 | 等闭源业务开发跑出足够真实样本后再评估 | 不做“先把多 Agent 平台、持续学习系统一次做完” |
| 前端与独立面板 | 继续承认当前可控边界只有官方 Codex 面板的入口层、脚本层、真源层 | 若将来真的拥有官方可控 UI 源码，再另立项目评估 | 不在当前仓强做独立前端面板，不把丞相模式误做成另一个产品壳 |
| 跨 harness / 跨平台 | 当前不作为近期目标 | 等真实出现分发需求后再单独评估 | 不为了“看起来更强”提前铺多 harness、大兼容层 |
| 仓库卫生与本地噪音 | 继续把运行态、本地噪音、证据稿和现行标准件分开；`.codex/chancellor/.rate-limit-state.json` 继续保持未跟踪 | 以后可继续把其他本地噪音文件逐个清点并归类 | 不把运行态缓存、日志、速率状态文件纳入公开仓跟踪 |

## 当前阶段必须做到位的收口线

1. `传令：XXXX` 主链必须长期稳定，不允许今天能用、明天漂移。
2. `VERSION.json`、`manifest.json`、运行态版本镜像、安装记录必须继续保持一致。
3. 普通用户对维护层只记 4 个动作，不再扩大记忆负担。
4. 任务收口、状态回写、真实状态总表必须能支撑你日常开发闭源业务项目。
5. 丞相模式的新增工作，必须证明它能直接提高你真实开发效率，而不是只让系统看起来更大。

## 允许重开边界的条件

只有同时满足下面条件时，才建议重新讨论“要不要扩丞相模式能力面”：

1. 现有单入口主链已经稳定跑过一段真实业务开发周期。
2. 真实任务包里反复出现同一类高频痛点，而不是偶发想法。
3. 拟新增能力不会破坏 `传令：XXXX` 这个唯一入口。
4. 拟新增能力不会越权改用户全局配置，也不会增加明显维护负担。
5. 这项扩展比直接开发闭源业务本身更划算。

## 对闭源业务主线的执行建议

| 问题 | 当前建议 |
| --- | --- |
| 要不要等丞相模式“完整做完”再做闭源业务项目 | 不要 |
| 丞相模式现在的定位是什么 | 开发控制面与稳态底盘 |
| 现在最该做什么 | 用当前稳定主链直接开始和持续推进闭源业务开发 |
| 丞相模式后续怎么演进 | 只从真实业务任务里反向抽取高频复用件，少量、克制、可验证地补 |

## 最终拍板

```text
丞相模式当前封顶目标：
做成稳定、克制、可信的开发控制面。

丞相模式当前明确不追：

```
