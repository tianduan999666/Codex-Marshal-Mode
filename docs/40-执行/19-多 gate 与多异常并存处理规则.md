# 多 gate 与多异常并存处理规则

最后更新：2026-03-29
状态：现行维护标准件
适用范围：当前仓复杂任务包的并发阻塞判断
说明：本规则用于在一个任务同时存在多个待拍板事项、多个异常记录、或拍板与异常并存时，给出统一的主阻塞判定、状态落盘与汇报顺序，避免口径分裂。

## 一句话结论

一个任务可以同时有多个待处理事项，但 `state.yaml` 只能有一个主状态；当多 gate 与多异常并存时，必须先按本规则选出主阻塞，再决定 `state.yaml`、`result.md` 与对主公的汇报口径。

## 第一原则：单任务单主状态

- `state.yaml` 永远只表达一个当前主状态。
- 其他未解决事项继续分别留在 `gates.yaml`、`result.md`、`decision-log.md`，而不是争抢 `state.yaml`。
- 若同时存在多个阻塞，必须选出“当前最先阻断推进”的主阻塞。

## 第一原则：单任务单主状态固定槽位

- `state.yaml 单主状态`：`state.yaml` 永远只表达一个当前主状态。
- `次要事项留档`：其他未解决事项继续分别留在 `gates.yaml`、`result.md`、`decision-log.md`，而不是争抢 `state.yaml`。
- `主阻塞唯一`：若同时存在多个阻塞，必须选出“当前最先阻断推进”的主阻塞。

## 第二原则：按下一行动主体决定优先级

当多个阻塞并存时，按“谁是下一行动主体”确定主状态：

1. `waiting_assist`：下一步必须由外部协助、额外信息或环境修复先发生。
2. `waiting_gate`：下一步必须由主公拍板先发生。
3. `paused`：当前执行者必须先手动停住，不允许继续推进。
4. `ready_to_resume`：已具备恢复条件，只差按既定恢复点继续。
5. `running / ready / verifying / done`：无主阻塞，进入正常推进态。

## 第三原则：多个 gate 并存时的主次划分

若 `gates.yaml` 中同时存在多个 `pending` gate：

- 只选一个“主 gate”进入当前汇报口径。
- 主 gate 优先级按以下顺序判断：
  1. 直接阻断当前下一步的 gate
  2. 影响范围最大的 gate
  3. 截止时间最早的 gate
- 未被选为主 gate 的事项继续保留在 `gates.yaml`，并在 `result.md` 中列为“次要待处理项”。

## 第四原则：gate 与异常并存时的裁决

- 若异常导致当前连拍板包都不具备讨论条件，主状态优先用 `waiting_assist` 或 `paused`，gate 保留但不主导当前状态。
- 若异常已收口，当前仍需主公拍板才能继续，主状态用 `waiting_gate`。
- 若主公已拍板，但恢复动作尚未准备好，主状态按异常结果切到 `paused` 或 `ready_to_resume`。
- 若拍板结果本身要求回退，先完成 `decided/dropped` 回写，再按异常模板切换到新的异常态。

## 第五原则：文档落盘分工

- `state.yaml`：只写主状态、主下一步。
- `gates.yaml`：保留全部 gate 项及其状态。
- `decision-log.md`：记录为什么本轮选择该主状态，而不是其他状态。
- `result.md`：写清“主阻塞”“次要待处理项”“恢复顺序”。

## 第五原则：文档落盘分工固定槽位

- `state.yaml`：只写主状态、主下一步。
- `gates.yaml`：保留全部 gate 项及其状态。
- `decision-log.md`：记录为什么本轮选择该主状态，而不是其他状态。
- `result.md`：写清“主阻塞”“次要待处理项”“恢复顺序”。

## 推荐汇报顺序

1. 先给主状态结论。
2. 再说明主阻塞是谁、为什么它优先。
3. 再列次要待处理项。
4. 最后说明一旦主阻塞解除，恢复顺序是什么。

## 推荐汇报顺序固定槽位

- `主状态结论`：先给主状态结论。
- `主阻塞理由`：再说明主阻塞是谁、为什么它优先。
- `次要待处理项`：再列次要待处理项。
- `恢复顺序`：最后说明一旦主阻塞解除，恢复顺序是什么。

## 收口检查

- 是否已明确主状态为何不是其他候选状态。
- 是否已把次要 gate 或异常留在对应文件中，没有被抹掉。
- 是否已在 `result.md` 中写明主阻塞与次要待处理项。
- 是否已在 `decision-log.md` 中记录本轮裁决依据。
- 是否已在恢复后重新评估主状态，而不是沿用旧状态。

## 收口检查固定槽位

- `主状态排他性`：是否已明确主状态为何不是其他候选状态。
- `次要事项保留`：是否已把次要 gate 或异常留在对应文件中，没有被抹掉。
- `result.md 完整性`：是否已在 `result.md` 中写明主阻塞与次要待处理项。
- `decision-log.md 留痕`：是否已在 `decision-log.md` 中记录本轮裁决依据。
- `恢复后重评`：是否已在恢复后重新评估主状态，而不是沿用旧状态。

## 与现有模板的关系

- 单个待拍板事项：优先使用 `docs/40-执行/16-拍板包半自动模板.md`
- 单个拍板结果回写：优先使用 `docs/40-执行/17-拍板结果回写模板.md`
- 单个异常或回退：优先使用 `docs/40-执行/18-异常路径与回退模板.md`
- 若多个 gate 或多个异常并存，先按本文裁决主状态，再决定用哪个单项模板继续处理。
- 若已裁决主状态并需要快速统一汇报骨架，继续使用 `docs/40-执行/20-复杂并存汇报骨架模板.md`。

## 本文档的价值

- 把复杂任务下的主阻塞选择从临场判断变成固定规则。
- 降低多事项并存时状态口径漂移的概率。
- 为后续是否做多 gate 自动裁决保留稳定规则底座。
