# V4 业务开发阶段最短作战手册

最后更新：2026-04-05
状态：现行执行件
适用范围：使用 `丞相模式` 开发闭源业务项目时的日常最短打法
说明：本手册只回答“现在怎么用丞相模式干业务开发最省脑子”，不展开维护层细节。

## 一句话结论

不要等丞相模式“完整做完”再开发闭源业务项目。当前最优打法是：把丞相当成稳定开发控制面，直接在官方 Codex 面板里用 `传令：XXXX` 推业务。

## 先记住的原则

1. 丞相模式当前定位是“开发控制面”，不是闭源业务项目本身。
2. 闭源业务推进优先级，高于继续把丞相模式扩成大平台。
3. 日常开发优先在官方 Codex 面板里完成，不主动把自己拉回终端。
4. 真正重复出现的高频动作，才值得回收进丞相模式；偶发想法不值得平台化。

## 最短作战链

```text
先确认版本与状态
  ↓
直接下达自然语言任务
  ↓
让丞相执行、收口、留痕
  ↓
下一轮直接继续当前任务
```

## 开工前怎么做

| 你现在要干什么 | 你对丞相说什么 | 目的 |
| --- | --- | --- |
| 先看现在稳不稳 | `传令：状态` | 确认当前运行态、自检状态、当前任务 |
| 先看是不是最新版本 | `传令：版本` | 确认现在跑的是哪版丞相 |
| 直接开始新需求 | `传令：修一下登录页` | 让丞相按当前单入口主链直接开工 |
| 续做昨天那一刀 | `传令：继续当前任务` | 优先接上已有激活任务 |
| 准备换到新聊天 | `传令：交班` | 为当前任务生成交班单与进度快照 |
| 新聊天无缝续上 | `传令：接班` | 优先读取交班单，不要求重述背景 |

## 开发闭源业务时的推荐说法

下面这些说法最适合你现在的业务开发阶段：

```text
传令：修一下登录页
传令：把订单列表的筛选逻辑补完整
传令：继续当前任务
传令：交班
传令：接班
传令：这一刀已完成，请收口并落盘
```

不建议的说法：

```text
先别干活，先把丞相模式做成 36 个 Agent 平台
先把所有未来会用到的技能库一次做完
先把独立前端面板做出来再说
```

原因很简单：这些都在挤占闭源业务主线。

## 一轮业务开发的标准节奏

| 阶段 | 你怎么说 | 预期结果 |
| --- | --- | --- |
| 开工 | `传令：修一下登录页` | 丞相必要时先轻检、修平，再进入任务执行 |
| 中途续做 | `传令：继续当前任务` | 接上已有任务，不重复新建 |
| 准备切新聊天 | `传令：交班` | 生成 `progress-snapshot.md` 与 `handoff.md` |
| 新聊天接上 | `传令：接班` | 优先读取交班单，直接说清背景、做法与下一步 |
| 中途核对 | `传令：状态` | 看到版本、上次检查、当前任务等固定 6 行 |
| 收口 | `传令：这一刀已完成，请收口。` | 按 `已完成 / 结果 / 下一步` 整理并留痕 |
| 要发版前自查 | 维护层执行 `%USERPROFILE%\.codex\self-check.cmd -MaintainerMode` | 一次看全漂移，只在维护时使用 |

## 什么情况下才回维护层

只有下面几种情况，才建议暂时离开面板回维护层：

1. `传令：XXXX` 一句话入口失败。
2. 你需要升级、自检、回滚。
3. 你怀疑运行态和真源有多处漂移，需要一次看全。
4. 你要正式提交、推送前，想确认治理门禁与版本纪律。

维护层普通只记这 4 个动作：

```powershell
.\install.cmd
%USERPROFILE%\.codex\upgrade.cmd
%USERPROFILE%\.codex\self-check.cmd
%USERPROFILE%\.codex\rollback.cmd
```

## 当前阶段不要做的 3 件事

| 不要做什么 | 原因 |
| --- | --- |
| 不要等丞相模式“全做完”再干闭源业务项目 | 这会让主线永久后移 |
| 不要把偶发需求立刻平台化 | 会制造平台幻觉和维护负债 |
| 不要把业务开发日常重新拉回终端 | 会破坏当前最值钱的面板单入口体验 |

## 当前阶段最值得追的 3 件事

| 该追什么 | 为什么 |
| --- | --- |
| 单入口稳定 | 这是日常效率最高的地方 |
| 收口和留痕稳定 | 这决定你能不能连续推进业务开发 |
| 从真实业务任务里反向提炼少量高频复用件 | 这是丞相模式后续演进的正路 |

## 最终执行建议

```text
闭源业务开发优先。
丞相模式只做稳定底盘。
先用，再从真实使用里反向提炼。
```
