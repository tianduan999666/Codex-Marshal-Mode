# 异常路径与回退模板

最后更新：2026-03-29
状态：现行维护标准件
适用范围：当前仓维护层异常、失败、撤销与回退动作
说明：本模板用于在任务推进出现失败、阻塞或需要回退时，半自动把状态切到 `paused`、`ready_to_resume` 或 `waiting_assist`，并把异常原因、回退范围、恢复提示沉淀进任务包。

## 一句话结论

当任务不能继续按正常链路推进时，优先使用当前仓内的异常路径与回退模板，而不是只在聊天里说明“先停一下”。

## 脚本位置

- `.codex/chancellor/record-exception-state.ps1`

## 适用场景

- 当前动作失败，需暂停并保留恢复点。
- 当前改动需要回退到上一稳定状态。
- 当前任务需等待外部协助或额外信息。
- 需要把异常原因、回退范围、恢复提示写入运行态。

## 输入项

- `TaskId`
- `ExceptionType`（`failed / rolled_back / blocked`）
- `Summary`
- `RollbackScope`
- `ResumeHint`
- `NextStatus`（`paused / ready_to_resume / waiting_assist`）
- `NextAction`
## 输出结果

脚本会同步写入：

- `state.yaml`：切换到异常后的真实状态，并更新 `next_action`
- `decision-log.md`：追加异常或回退记录
- `result.md`：追加异常路径摘要

## 使用方式

这是维护层动作，建议在发现当前链路无法继续时立刻执行。

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\.codex\chancellor\record-exception-state.ps1 `
  -TaskId v4-trial-019-example-task `
  -ExceptionType failed `
  -Summary "当前方案验证失败，需要暂停" `
  -RollbackScope "撤回本轮未验证通过的方案分支" `
  -ResumeHint "待补充新证据后，从 state.yaml 的 next_action 恢复" `
  -NextStatus paused `
  -NextAction "整理失败原因并准备下一轮恢复"
```

## 边界说明

- 当前脚本不会自动修改 `gates.yaml`。
- 当前脚本不会自动提交或推送。
- 当前脚本不会替代人工判断异常级别，只负责标准化落盘。
- 若异常与多个 gate 并存，先按 `docs/40-执行/19-多 gate 与多异常并存处理规则.md` 判断主状态。
## 生成后的动作

1. 回看 `state.yaml`，确认状态已切换到正确的异常态。
2. 回看 `decision-log.md` 与 `result.md`，确认异常原因、回退范围与恢复提示已经对齐。
3. 若后续可恢复，按 `docs/40-执行/05-跨轮恢复说明.md` 回到恢复流程。
4. 按 `docs/40-执行/14-维护层动作矩阵与收口检查表.md` 做统一收口。

## 长期价值

- 降低失败链路只留在聊天里的风险。
- 让 `paused / ready_to_resume / waiting_assist` 三类状态有统一落盘模板。
- 为后续异常路径自动恢复或回退策略保留稳定接口。
