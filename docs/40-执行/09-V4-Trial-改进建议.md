# V4-Trial 改进建议

最后更新：2026-03-29
状态：现行验收材料
适用阶段：`V4-Trial` 之后

## 一句话结论

下一阶段不宜立刻扩功能，应该先补‘稳定性、脚手架、归档机制’三件长期收益更高的事。

## 优先级 A：先补稳定性

### 1. 处理 `git index.lock` 问题

- 建议：当前已补 `docs/40-执行/10-本地安全提交流程.md`，下一阶段若仍频发，再考虑更工程化的串行提交机制。
- 价值：先把处置方案标准化，再视频率决定是否进一步自动化。

### 2. 增加 Trial 汇总入口

- 建议：把验收报告、缺陷清单、改进建议加入现行标准件总览与执行区入口。
- 价值：主公拍板与后续复盘更直接。

## 优先级 B：再补脚手架

### 3. 继续把半自动起包推进到更稳的脚手架

- 建议：当前已补半自动起包脚手架；下一阶段若继续推进，可考虑增加输入校验增强或更稳的起包命令包装。
- 价值：在已有过渡方案基础上继续减少重复劳动，进一步降低任务包格式漂移风险。

## 优先级 C：再补归档机制

### 4. 持续执行证据稿归档规则

- 建议：当前已补归档规则并完成第一批迁移；下一阶段重点从‘建立规则’转为‘持续执行与定期清理’。
- 价值：保持执行区长期清爽，避免证据稿再次长期堆积。

## 优先级 D：再决定是否升格

### 5. 以拍板制进入 `V4-Target`

- 建议：不要自动默认升格，先由主公根据验收报告与缺陷清单拍板。
- 价值：避免在缺陷未定级时过早扩展目标。

## 建议的下一步顺序

1. 同步 Trial 验收入口
2. 主公拍板 Trial 是否通过
3. 处理必须先修的缺陷
4. 再决定是否进入 `V4-Target`
