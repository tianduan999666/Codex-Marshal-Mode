# ECC 功能采纳建议与实施路线图

最后更新：2026-04-03
状态：现行规划文档
适用范围：决定“借鉴 ECC 什么，不借鉴什么”时使用

## 一句话结论

不要整包照抄 ECC。
只借它真正补短板的能力；凡是会打乱 `传令` 单入口、会增加学习成本、会越权改用户全局环境的，一律不采。

## 采纳决策表

| 项目 | 结论 | 原因 |
| --- | --- | --- |
| 多 Agent 运行时 | 暂不直接接 | 这是大工程，不是当前最小收益点 |
| Skills 体系 | 借鉴思路，不整包搬 | 当前先补“高频、通用、可验证”的少量能力更值钱 |
| 持续学习 / Instincts | 保留为后续方向 | 有价值，但不是当前卡点 |
| Hooks 自动化 | 谨慎借鉴 | Codex 侧官方能力边界仍有限，强推容易假繁荣 |
| 跨 harness 兼容 | 暂不优先 | 你当前主线是官方 Codex 面板，不是多平台分发 |
| 单入口协议 | 继续坚持 | `传令：XXXX` 已经是当前最有产品感的核心资产 |

## 现在最该补的 3 件事

| 优先级 | 项目 | 当前状态 | 为什么值钱 |
| --- | --- | --- | --- |
| P0 | 高质量过程复用件 | 已有最小过程金句与收口骨架，但还没有形成“可扩可管”的能力包 | 这是最接近 Skills 但不增加心智成本的做法 |
| P0 | 面板内真实结果回写与收口模板固化 | 已有骨架 | 能直接提升日常可用性和一致性 |
| P1 | 从任务包历史提炼常见模式 | 未做 | 这是以后做“轻量学习”的最小起点 |

## 明确不做的事

| 不做项 | 原因 |
| --- | --- |
| 不做“36 个 Agent 幻觉路线图” | 当前仓并没有这套运行时，写进去只会自欺 |
| 不做“147 个 Skills 覆盖率”这类估算表 | 当前没有可靠依据，写了就是假精确 |
| 不做“独立面板” | 你当前没有前端源码，也没有这个必要 |
| 不把用户入口重新扩成多命令体系 | 会直接破坏现在最值钱的单入口心智 |
| 不让安装器再碰用户全局 provider | 这是已经踩过的真实坑，不能回退 |

## 现实路线图

```text
现在
  ↓
稳住传令单入口 + 真源状态栏 + 安装稳态
  ↓
把高频过程能力做成少量、可管、可验证的复用件
  ↓
再考虑轻量学习
  ↓
最后才评估多 Agent 或更大自动化运行时
```

## 下一阶段建议

| 阶段 | 建议动作 | 目标 |
| --- | --- | --- |
| 近 1 轮 | 继续压缩对外文案与维护入口 | 保持简单 |
| 近 2-3 轮 | 把任务收口、回写、模板留痕做得更稳 | 提升日常复用效率 |
| 中期 | 从真实任务包里提炼“高频动作库” | 为轻量 Skills / 学习机制铺路 |
| 远期 | 再判断是否值得引入多 Agent runtime | 避免提前做重系统 |

## 最终原则

1. 先稳住你已经天天会用的主链。
2. 只补真实卡点，不补想象中的大而全。
3. 任何新能力都不能破坏 `传令：XXXX` 这个唯一入口。
4. 任何安装升级动作都不能再越权改用户全局模型配置。
