# ECC 功能采纳建议与实施路线图

最后更新：2026-04-02
状态：现行规划文档
适用范围：丞相模式功能增强规划

## 一、核心结论

基于《丞相模式与ECC功能对比分析》，建议采纳 ECC 的 3 个核心能力，保持丞相模式的独特优势，形成"严格质量控制 + 专业分工协作"的混合模式。

## 二、必须采纳（优先级 P0）

### 2.1 Agent 编排系统

**采纳理由**：
- 当前最大短板：单一 AI 实例无法并行处理、无专业化分工
- 直接影响效率：无法同时进行代码审查、安全扫描、测试验证
- 扩展性受限：无法针对特定场景定制专业 Agent

**采纳方案**：
```
阶段 1：集成 ECC 核心 4 个 Agent（1 周）
- planner：需求拆解和风险评估
- tdd-guide：测试驱动开发
- code-reviewer：代码质量审查
- security-reviewer：安全漏洞扫描

阶段 2：适配任务包结构（3 天）
- Agent 输出写入任务包对应槽位
- 保持任务包为唯一运行态
- 支持跨轮恢复

阶段 3：扩展到 10 个 Agent（2 周）
- architect：架构决策
- build-error-resolver：构建错误修复
- e2e-runner：端到端测试
- refactor-cleaner：死代码清理
- doc-updater：文档更新
- performance-optimizer：性能优化
```

**集成策略**：
- 保持丞相的反屎山总纲为最高约束
- Agent 输出必须通过质量门禁
- Agent 结果记录到任务包 decision-log

**预期收益**：
- 效率提升 3-5 倍（并行处理）
- 质量更稳定（专业 Agent 保证）
- 可扩展性增强（按需添加 Agent）

### 2.2 测试规范与 TDD 工作流

**采纳理由**：
- 当前缺失：反屎山总纲要求 80% 覆盖率，但无具体测试规范
- 质量风险：无 TDD 工作流指导，测试质量不稳定
- 团队协作：无统一测试标准

**采纳方案**：
```
阶段 1：补充测试规范文档（3 天）
- 参考 ECC 的 tdd-guide Agent 能力
- 编写 docs/reference/03-测试规范与TDD工作流.md
- 明确测试框架选型标准（单元/集成/E2E）

阶段 2：集成到任务包模板（2 天）
- 在任务包模板中增加测试阶段
- 固化 TDD 流程：写测试 → 跑失败 → 写代码 → 跑通过 → 重构
- 在 contract.yaml 中增加测试覆盖率要求

阶段 3：开发测试辅助命令（1 周）
- 传令：测试 - 运行测试套件
- 传令：覆盖率 - 检查覆盖率
- 传令：TDD - 启动 TDD 工作流
```

**集成策略**：
- 测试规范纳入反屎山总纲
- 测试覆盖率纳入质量门禁
- 测试结果记录到任务包 result.md

**预期收益**：
- 测试质量标准化
- 减少线上 Bug
- 提升代码可维护性

### 2.3 工作流自动化

**采纳理由**：
- 当前痛点：任务包各阶段需要手动触发，容易遗漏
- 效率损失：重复执行相同步骤
- 一致性差：不同任务执行流程可能不同

**采纳方案**：
```
阶段 1：开发 `传令：收口` 命令（1 周）
- 自动执行：测试 → 代码审查 → 安全扫描 → 提交 → 归档
- 任何环节失败则中断并给出修复建议
- 全部通过后自动提交并更新任务包状态

阶段 2：固化标准流程（3 天）
- 在任务包模板中固化 5 阶段流程
- 规划 → 开发 → 测试 → 审查 → 收口
- 每个阶段有明确的输入输出和验收标准

阶段 3：开发流程编排引擎（2 周）
- 支持自定义流程编排
- 支持条件分支（如：测试失败 → 修复 → 重测）
- 支持并行执行（如：代码审查 + 安全扫描同时进行）
```

**集成策略**：
- 保持任务包为流程状态载体
- 流程编排结果记录到 state.yaml
- 支持流程中断和恢复

**预期收益**：
- 减少人工干预 60%
- 降低遗漏风险
- 提升流程一致性

## 三、建议采纳（优先级 P1）

### 3.1 持续学习机制

**采纳理由**：
- 长期价值：自动积累项目经验，减少重复错误
- 维护成本：降低规则文档人工维护成本
- 团队协作：自动共享最佳实践

**采纳方案**：
```
阶段 1：设计 Instincts 存储结构（1 周）
- 在 .codex/chancellor/instincts/ 记录项目模式
- 格式：YAML，包含模式描述、置信度、来源任务包
- 分类：编码模式、架构模式、测试模式、文档模式

阶段 2：开发 `传令：学习` 命令（2 周）
- 分析任务包历史提取规律
- 识别重复出现的模式（如：特定错误的修复方式）
- 生成 Instinct 文件并标注置信度

阶段 3：集成到执行流程（1 周）
- 任务开始前加载相关 Instincts
- 执行过程中应用已学习的模式
- 任务结束后更新 Instincts
```

**集成策略**：
- Instincts 优先级低于反屎山总纲
- 人工可以审查和修正 Instincts
- 支持导出和共享 Instincts

**预期收益**：
- 减少重复错误 70%
- 降低规则维护成本 50%
- 提升 AI 理解项目能力

### 3.2 团队协作能力

**采纳理由**：
- 扩展场景：从单人工具升级为团队工具
- 标准化：统一团队编码风格和流程
- 降低成本：减少新人上手时间

**采纳方案**：
```
阶段 1：开发 `传令：提取规范` 命令（1 周）
- 参考 ECC 的 /skill-create
- 分析 Git 历史提取团队规范
- 生成 .codex/chancellor/team-rules.md

阶段 2：支持多人共享任务包（2 周）
- 任务包支持多人协作标记
- 记录每个阶段的执行人
- 支持任务交接和状态同步

阶段 3：开发团队仪表盘（3 周）
- 展示团队任务包统计
- 展示团队质量指标（覆盖率、缺陷率）
- 展示团队规范遵守情况
```

**集成策略**：
- 团队规范纳入反屎山总纲
- 任务包支持团队协作元数据
- 保持单人使用体验不变

**预期收益**：
- 支持 3-5 人小团队协作
- 统一团队标准
- 降低新人上手成本 60%

## 四、暂不采纳（优先级 P2）

### 4.1 完整 Skills 体系（147 个）

**暂不采纳理由**：
- 当前覆盖度已达 35%，满足基本需求
- 维护成本高：147 个 Skills 需要持续维护
- 边际收益递减：后续 Skills 使用频率低

**替代方案**：
- 按需补充高频 Skills（预计 20-30 个）
- 优先补充测试、部署、监控类 Skills
- 建立 Skills 贡献机制，社区驱动

### 4.2 独立前端 UI

**暂不采纳理由**：
- 当前面板入口已满足需求
- 开发成本高，维护成本高
- 与"保持官方面板为默认入口"的设计哲学冲突

**替代方案**：
- 优化面板入口体验
- 开发 VSCode 插件（轻量级）
- 提供 CLI 命令行入口（高级用户）

## 六、风险与应对

### 6.1 集成复杂度风险

**风险**：ECC Agent 系统与丞相任务包结构不兼容

**应对**：
- 先做小范围试点（1-2 个 Agent）
- 设计适配层，隔离 ECC 和丞相的差异
- 保持任务包结构稳定，Agent 输出适配任务包

### 6.2 质量控制风险

**风险**：引入 Agent 后质量标准降低

**应对**：
- 反屎山总纲保持最高优先级
- Agent 输出必须通过质量门禁
- 人工可以否决 Agent 建议

### 6.3 维护成本风险

**风险**：Agent 系统维护成本高

**应对**：
- 优先集成成熟 Agent，减少自研
- 建立 Agent 测试套件
- 社区驱动 Agent 维护

## 八、总结

**核心策略**：采纳 ECC 的专业分工和自动化能力，保持丞相的严格质量控制和任务持久化优势，形成"严格约束 + 高效协作"的混合模式。

**优先级排序**：
1. P0：Agent 编排、测试规范、工作流自动化（4 周）
2. P1：持续学习、团队协作（6 周）
3. P2：完整 Skills、独立前端（按需）

**预期收益**：
- 效率提升 3-5 倍
- 质量更稳定
- 支持团队协作
- 降低维护成本

1. 评估 ECC Agent 系统集成可行性
2. 设计 Agent 与任务包的适配层
3. 启动 Phase 1 开发（4 周冲刺）
