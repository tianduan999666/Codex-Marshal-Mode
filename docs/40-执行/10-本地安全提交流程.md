# 本地安全提交流程

最后更新：2026-03-29
状态：现行标准件
适用范围：当前仓的本地提交流程
说明：本文件用于降低 `.git/index.lock` 冲突概率，作为当前仓的默认提交流程说明

## 一句话结论

当前仓执行提交时，应采用串行提交、串行推送的固定顺序，避免并发操作造成 `.git/index.lock` 冲突。

## 适用场景

- 文档连续迭代
- 多轮小步推进
- 需要边改边提交的 Trial 或 Target 阶段

## 推荐流程

1. 先确认待提交范围，只保留公开安全文件进入提交面。
2. 先执行 `git add`，不要与 `git commit` 并发触发。
3. 确认暂存区正确后，再单独执行 `git commit`。
4. 提交完成后，再单独执行 `git pull --rebase origin main`。
5. Rebase 成功后，再单独执行 `git push origin main`。

## 禁止动作

- 不要把 `git add` 与 `git commit` 并行执行。
- 不要把 `git commit` 与 `git push` 并行执行。
- 不要在存在锁文件疑虑时继续重复提交命令。

## 遇到 `index.lock` 的处置步骤

1. 先确认是否真的存在 `.git/index.lock`。
2. 若存在，先停止新的 Git 命令，不连续重试。
3. 确认没有其他 Git 进程仍在运行后，再清理锁文件。
4. 清理后按“`git commit` → `git pull --rebase` → `git push`”的串行顺序继续。

## 当前默认约定

- 当前仓默认采用串行 `git` 提交流程。
- 若后续引入脚手架或自动化，也必须保持提交阶段串行执行。
- 若再次出现锁文件问题，应先按本文件处置，再决定是否需要进一步工程化修复。

## 与缺陷清单的关系

- 本文件用于把 `git index.lock` 问题从“纯临时绕开”升级为“已有现行处置方案”。
- 该问题在 `V4-Trial` 阶段仍可保留为已知缺陷，但不再是完全无解状态。
