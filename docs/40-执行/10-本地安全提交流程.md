# 本地安全提交流程

最后更新：2026-04-05
状态：现行标准件
适用范围：当前仓的本地提交流程
说明：本文件用于降低 `.git/index.lock` 冲突概率，并把公开提交前的治理预检与本地硬门禁收入口当前仓的默认提交流程

## 一句话结论

当前仓执行提交时，应采用串行提交、串行推送的固定顺序，避免并发操作造成 `.git/index.lock` 冲突。

## 适用场景

- 文档连续迭代
- 多轮小步推进
- 需要边改边提交的 Trial 或 Target 阶段

## 推荐流程

1. 先确认待提交范围，只保留公开安全文件进入提交面。
2. 若当前轮影响现行标准件、公开口径或关键边界，先按 `docs/30-方案/08-V4-治理审计候选规范.md` 完成治理审计复核，再按 `docs/40-执行/21-关键配置来源与漂移复核模板.md` 完成配置来源与漂移复核。
3. 确保本地已安装公开提交治理硬门禁；默认通过 `.git/hooks/pre-push` 自动触发，不再依赖人工临场提醒。
4. 先执行 `git add`，不要与 `git commit` 并发触发。
5. 确认暂存区正确后，再单独执行 `git commit`。
6. 提交完成后，再单独执行 `git pull --rebase origin main`。
7. Rebase 成功后，再单独执行 `git push origin main`；推送时本地 `pre-push` hook 会自动执行 `.codex/chancellor/invoke-public-commit-governance-gate.ps1`。

## 提交前最小预检

- 当前轮若改动了现行标准件，先补一次治理审计复核。
- 当前轮若需要解释配置来源、版本依据或漂移判断，先补一次配置来源与漂移复核。
- 当前轮若改动了 `codex-home-export/manifest.json` 的 `included` 所覆盖的任一受管文件，先同步 bump `codex-home-export/VERSION.json`，再继续提交。
- 当前轮若刚改完 `codex-home-export` 受管文件，先重装或重同步运行态，并重新通过验真；不要把“同版本不同内容”直接推上去。
- 当前轮若涉及未公开闭源项目，公开文档、速查卡、任务标题、提交信息、推送说明统一只写 `业务项目`；只有在必须强调保密属性时才写 `闭源项目`，禁止带出真实项目名、内部代号或可反推身份的专有称呼。
- 当前轮若仍有 `.codex/`、`logs/`、`temp/generated/`、`.vscode/`、`.serena/` 等运行态或本地工具状态混入提交面，先清提交范围，再继续。
- 当前轮若看到 `.codex/chancellor/.rate-limit-state.json` 出现在 `git status`，应把它视为本地噪音并留在未跟踪态，不作为公开提交对象。
- 当前轮若入口文档之间口径未修平，先修平，再提交。
- 当前轮若尚未安装本地 `pre-push` 门禁，先执行 `.codex/chancellor/install-public-commit-governance-hook.ps1`，再继续。

## 禁止动作

- 不要把 `git add` 与 `git commit` 并行执行。
- 不要把 `git commit` 与 `git push` 并行执行。
- 不要在未完成必要治理复核时直接提交公开口径改动。
- 不要绕过本地 `pre-push` 门禁直接推送公开改动。
- 不要在存在锁文件疑虑时继续重复提交命令。

## 遇到 `index.lock` 的处置步骤

1. 先确认是否真的存在 `.git/index.lock`。
2. 若存在，先停止新的 Git 命令，不连续重试。
3. 确认没有其他 Git 进程仍在运行后，再清理锁文件。
4. 清理后按“`git commit` → `git pull --rebase` → `git push`”的串行顺序继续。

## 当前默认约定

- 当前仓默认采用串行 `git` 提交流程。
- 当前仓默认把“治理审计复核 + 配置来源与漂移复核”视为公开提交前的前置检查，而不是事后补写。
- 当前仓默认通过本地 `pre-push` hook 自动执行公开提交治理门禁，不再依赖人工提醒。
- 若后续引入脚手架或自动化，也必须保持提交阶段串行执行，并继续经过当前门禁。
- 若再次出现锁文件问题，应先按本文件处置，再决定是否需要进一步工程化修复。
