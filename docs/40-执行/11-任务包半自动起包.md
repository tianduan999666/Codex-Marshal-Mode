# 任务包半自动起包

最后更新：2026-04-01
状态：现行标准件
适用范围：维护层起包动作
说明：这是维护层能力，用于减少新任务包 5 件套的重复创建成本；普通使用仍以面板为主

## 一句话结论

当需要新建任务包时，优先使用当前仓内的半自动起包脚手架，而不是手工重复创建 5 个文件。

## 脚手架位置

- `.codex/chancellor/create-task-package.ps1`
- `codex-home-export/new-task.ps1`（面向 Windows 的轻入口；默认自动分配下一个 `v4-target-*` 任务编号，也可显式选择 `trial` 或 `target`，并给出回到官方 Codex 面板的粘贴话术）

## 适用场景

- 默认新建 `v4-target-<三位序号>-<语义名>` 任务包
- 需要显式新建 `v4-trial-*` 或 `v4-target-*` 任务包
- 需要一次性生成 5 件套基础文件
- 需要同时更新 `active-task.txt`

## 输入项

- `TaskId`
- `Title`
- `Goal`
- `TaskNamespace`（轻入口可选；`trial / target`，默认 `target`）
- `PhaseHint`（可选）
- `PlanningHint`（可选；主假设）
- `PlanStep`（可选；当前最小推进步）
- `VerifySignal`（可选；当前步完成后要看什么信号）
- `InitialStatus`（可选；默认 `drafting`，轻入口会自动改为 `running`）
- `RiskLevel`（默认 `low`）
- `SetActiveTask`（默认启用）

## 输出结果

脚手架会在 `.codex/chancellor/tasks/<task-id>/` 下生成：

- `contract.yaml`（已带规划与治理来源入口）
- `state.yaml`（已带最小推进步与验证信号）
- `decision-log.md`（已带路线修正与治理提示骨架）
- `gates.yaml`
- `result.md`（已带规划复核与治理复核骨架）

## 使用方式

这是维护层动作，建议在需要批量或频繁起包时使用。

### 轻入口（更适合公开试用）

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\codex-home-export\new-task.ps1 -Title "修复登录页 bug"
```

轻入口会自动完成 4 件事：

1. 默认自动找出下一个 `v4-target-<三位序号>`
2. 自动根据标题生成语义任务名
3. 自动创建任务包 5 件套
4. 自动打印一段可直接贴回官方 Codex 面板的话术

如需显式创建 Trial 任务，可执行：

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\codex-home-export\new-task.ps1 -Title "补做 Trial 冒烟" -TaskNamespace trial
```

### 完整脚手架（更适合维护层精细控制）

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\.codex\chancellor\create-task-package.ps1 `
  -TaskId v4-target-004-example-task `
  -Title "示例任务" `
  -Goal "把示例任务的 5 件套骨架一次生成" `
  -PhaseHint example_phase `
  -PlanningHint "当前主假设是先固化任务边界再推进" `
  -PlanStep "补齐任务边界与验收" `
  -VerifySignal "contract.yaml 已补齐关键边界与验收"
```

如不希望切换当前激活任务，请使用：

```powershell
powershell.exe -Command "& '.\.codex\chancellor\create-task-package.ps1' `
  -TaskId 'v4-target-005-example-task' `
  -Title '示例任务' `
  -Goal '只创建骨架，不切换激活任务' `
  -PhaseHint 'example_phase' `
  -SetActiveTask $false"
```

## 生成后的动作

1. 回看 `contract.yaml`，补齐任务特定限制、验收标准与主假设。
2. 回看 `state.yaml`，把 `next_action`、`plan_step`、`verify_signal` 改成真实内容。
3. 回看 `decision-log.md` 与 `result.md`，确认路线修正、规划复核与治理复核骨架已可直接使用。
4. 按 `docs/30-方案/07-V4-规划策略候选规范.md` 判断当前是否需要改路。
5. 提交前按 `docs/30-方案/08-V4-治理审计候选规范.md` 追加治理审计复核。
6. 如当前轮涉及入口、现行标准件或公开口径，再按 `docs/40-执行/21-关键配置来源与漂移复核模板.md` 追加配置来源与漂移复核。
7. 按 `docs/40-执行/14-维护层动作矩阵与收口检查表.md` 完成本轮收口。
8. 如当前不应切换激活任务，可在创建后手动调整 `active-task.txt`。

## 边界说明

- 该脚手架只负责生成最小骨架，不替代任务判断。
- 该脚手架不会自动提交或推送。
- 该脚手架不会创建未批准目录。

## 长期价值

- 减少重复创建 5 件套文件的机械劳动。
- 降低任务包格式漂移风险。
- 让“起包”天然接上“收口”，减少只起包不闭环的半成品状态。
- 为后续是否做更强自动化提供低风险过渡层。
