# 跨轮恢复说明

最后更新：2026-03-29
状态：现行标准件
适用阶段：`V4-Trial`
说明：本文件用于说明如何仅依靠当前目录中的任务包文件，在新会话中恢复任务，不依赖长聊天历史

## 一句话目标

验证一条任务在暂停或跨轮后，能否仅靠任务包文件被重新理解、继续推进与复验。

## 恢复的最小前提

一个可恢复任务包至少应具备：

- `contract.yaml`
- `state.yaml`
- `decision-log.md`
- `gates.yaml`
- `result.md`

## 恢复检查清单

1. 先读取 `active-task.txt`，确认默认激活任务 ID。
2. 打开对应任务目录，确认 5 件套文件齐全。
3. 先读 `state.yaml`，获取当前状态、风险、下一步、阻塞。
4. 再读 `contract.yaml`，确认任务目标、限制与验收条件。
5. 再读 `decision-log.md`，确认之前已经做过哪些关键判断。
6. 视需要读 `gates.yaml`，确认是否存在待拍板事项。
7. 最后读 `result.md`，确认已有结果、验证证据与遗留事项。
8. 若同时存在多个 gate 或异常，先按 `docs/40-执行/19-多 gate 与多异常并存处理规则.md` 判断当前主状态。

## 恢复成功的判断标准

- 新会话中的执行者无需回翻旧聊天，也能说清当前任务状态。
- 能明确区分“已完成什么”“下一步做什么”“还缺什么”。
- 若任务处于 `paused` 或 `ready_to_resume`，能明确恢复点。
- 若任务已 `done`，能明确不再重复执行，而是转向下一任务。

## 恢复失败信号

- 只看任务包仍无法理解当前任务到底在做什么。
- `state.yaml`、`contract.yaml`、`result.md` 三者口径冲突。
- 任务虽然已完成，但恢复后仍要重新从头判断方向。
- 仍然必须依赖长聊天历史才能继续推进。

## 失败后的处置动作

1. 回补 `state.yaml` 的下一步与阻塞信息。
2. 回补 `result.md` 的验证证据与遗留事项。
3. 若关键决策缺失，补写 `decision-log.md`。
4. 若任务边界不清，回补 `contract.yaml` 的目标与验收。
5. 若需要把失败或回退结果标准化落盘，使用 `docs/40-执行/18-异常路径与回退模板.md`。

## 与 Trial 验收的关系

- 本文档用于支撑第三条真实试跑闭环。
- 若恢复无法独立成立，说明 `V4-Trial` 仍未真正脱离聊天记忆。
- 本文档可进入公开仓；真实运行态与日志继续只留本地。
