# V4-Target 实施计划

最后更新：2026-03-29
状态：现行准备稿
适用阶段：`V4-Target` 准备阶段
说明：本计划基于当前仓 Trial 验收结果与已落地治理项，给出下一阶段最优推进顺序

## 一句话结论

`V4-Target` 应按“稳态收口优先、自动化渐进、安装链后置”的顺序推进，而不是一开始就铺太大。

## 推荐推进顺序

### T1：冻结 Target 口径

- 固化 `V4-Target` 蓝图
- 固化 `V4-Target` 实施计划
- 把进入正式阶段前的拍板条件写清楚

### T2：稳定维护层动作

- 继续验证安全提交流程
- 继续使用半自动起包脚手架
- 固化维护动作矩阵与收口检查表
- 补必要的输入校验或说明增强

### T3：固化归档节奏

- 定期把失去现行职责的证据稿迁入 `docs/90-归档/`
- 保持执行区长期清爽

### T4：准备更强工程化能力

- 评估是否要把起包脚手架继续增强
- 评估是否需要更稳的提交包装或维护入口
- 评估哪些 Target 能力值得进入正式实现

### T5：再决定是否进入更大范围实现

- 再看是否需要更独立的安装或迁移能力
- 再看是否需要更完整的 Memory / Policy / Evolution 能力
- 保持‘先必要、后豪华’原则

## 每阶段统一验收口径

- 是否继续保持当前目录内自含
- 是否继续保持现行件与证据稿边界清楚
- 是否继续保持任务包可恢复、可追溯
- 是否减少维护层重复劳动，而不是增加新负担

## 当前最优下一步

1. 主公拍板是否认可进入 `V4-Target` 准备阶段
2. 若认可，则按 T1、T2、T3 顺序推进
3. T4 与 T5 保持后置，不抢跑

## 当前不建议立即做的事

- 不直接重构安装链
- 不直接扩独立 UI
- 不直接上重型自动化
- 不直接把旧仓资产整包搬入当前仓
