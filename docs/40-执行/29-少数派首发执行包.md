# 少数派首发执行包

最后更新：2026-04-02
状态：现行标准件
适用范围：第一轮公开试用的少数派首发
说明：目标不是继续改文字风格，而是给主公一份今天就能直接发的少数派定稿与跟进卡片

## 一句话结论

如果第一轮改成先发 `少数派`，就不要再临场改。直接用这一版标题和正文发出去；发完后只按固定动作记反馈、接回复、做 24 小时判断。

## 先记住这 4 条

1. 第一轮只测一个主渠道，先不分心。
2. 首帖先讲痛点和结果，不先讲内部体系。
3. 主入口始终写成官方 Codex 面板，不写回 Claude。
4. 帖子发出后，先收事实，不先急着改文案。

## 今天直接发这一版

### 标题

```text
我在做一套让 Codex 会话不再“失忆”的小工具，想邀请几位读者试用
```

### 正文

```text
这套工具想解决的不是“怎么多做一个按钮”，而是一个很烦的小问题：
每次打开官方 Codex 面板开新会话，常常还没开始干活，就得先重讲背景。

我现在把自己的这套继续工作流程整理成了公开试用版：
- 可以先初始化
- 可以再创建任务
- 然后直接回到 Codex 面板继续做
```
```text
目前先支持 Windows 开发者。
如果你愿意试，我可以免费帮你完成第一轮配置，也欢迎直接告诉我：你最怕哪一步最麻烦。
```

## 发出去前最后核对

- 有没有把入口写成官方 Codex 面板
- 有没有保留“Windows 开发者”这个筛选条件
- 有没有保留“欢迎直接告诉我哪一步最麻烦”
- 有没有避免把帖子写成过度技术说明

## 发帖后 30 分钟动作卡

1. 把发帖时间、链接、标题版本记进 `docs/40-执行/24-第一轮试用反馈记录表.md`
2. 把准备用的跟进话术对照 `docs/40-执行/25-试用接待与跟进话术.md` 再看一遍
3. 先不重写帖子，先等第一批真实回复

## 回复分流卡

### 对方说“我想试”

直接回：

```text
好呀，我这套现在先支持 Windows 开发者试用。
你如果方便，我先把仓库和最短上手路径发你：
1. 先跑初始化
2. 必要时验真一下
3. 再建一个任务，然后回官方 Codex 面板直接继续

你试的时候如果卡住，直接把报错或卡点发我就行，我会按第一轮试用帮你一起走通。
```

### 对方说“这个思路不错，但我还想再看看”

直接回：

```text
完全可以，这一轮我主要想先收第一手真实卡点。
你不用现在就决定，后面如果想试，直接回我一句就行；
我这边会继续把第一次上手路径压短。
```

### 对方明显不适配

直接回：

```text
感谢关注。
我这一轮先只收 Windows 开发者的真实试用反馈，
这样我能先把第一条上手路径压稳。
后面如果扩到别的平台，我再继续放开。
```

## 发帖后 24 小时动作卡

### 先看这 4 个事实

- 有没有人回复
- 有没有人明确说愿意试
- 有没有人已经开始走第一轮
- 最大卡点是在“没开始”还是“开始后卡住”

### 然后只做这 3 件事

1. 用 `25-试用接待与跟进话术.md` 轻跟进一次
2. 把真实原话和人数补进 `24-第一轮试用反馈记录表.md`
3. 不因为几句认可，就判断需求已经成立

## 这一刀之后接哪一份

1. 发帖当天看 `docs/40-执行/27-第一轮真实发帖执行清单.md`
2. 如果第一站先发少数派，就继续看本页
3. 有回复后看 `docs/40-执行/25-试用接待与跟进话术.md`
4. 需要沉淀事实时看 `docs/40-执行/24-第一轮试用反馈记录表.md`
5. 满 7 天后看 `docs/40-执行/26-第一轮试用周报模板.md`

## 最后提醒

- 少数派更看重叙述清楚，不要写成论坛式吆喝。
- 如果第一批回复偏少，先回看标题与首段，不要立刻回仓里大改系统。
- 如果已经有人愿意试，就先把人接住，这比继续补新文档更值钱。
