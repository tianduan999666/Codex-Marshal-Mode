# 拍板包准备与收口规范

最后更新：2026-03-29
状态：现行维护标准件
适用范围：当前仓全部需要主公拍板的任务
说明：本规范用于把待拍板事项从零散聊天结论，沉淀为可追溯、可比较、可收口的标准拍板包。

## 一句话结论

当任务进入 `waiting_gate` 或存在 `must_gate` 事项时，应先形成标准拍板包，再向主公汇报，不直接把半成品判断抛给主公。

## 什么时候必须准备拍板包

- `contract.yaml` 中 `must_gate` 不为空。
- `gates.yaml` 中存在待处理事项。
- 任务已进入 `waiting_gate`。
- 当前动作会突破边界、改变风险等级、改动现行口径或需要方案取舍。

## 拍板包最低组成

拍板包最少应覆盖以下四项：

1. 结论：一句话说明这次要拍什么。
2. 选项：给出可执行选项，数量控制在 2 到 3 个。
3. 影响：分别说明每个选项的收益、成本、风险与后续影响。
4. 建议：明确推荐项，并说明推荐理由。

## `gates.yaml` 推荐结构

```yaml
task_id: v4-trial-xxx-your-task
items:
  - gate_id: gate-001
    status: pending
    question: 是否允许执行某项边界变更
    recommendation: 选项 A
    options:
      - name: 选项 A
        impact: 短期成本较高，但长期复利更高
      - name: 选项 B
        impact: 短期更快，但后续维护成本更高
    needed_by: '2026-03-29 12:00:00'
    blocking_reason: 未拍板前不能继续下一步
```

## 字段填写原则

- `gate_id`：任务内唯一，不复用。
- `status`：统一使用 `pending / decided / dropped`。
- `question`：必须是可直接做 Yes/No 或多选决策的问题。
- `recommendation`：必须明确，不得留空。
- `options`：每个选项都要写清影响，避免只给标签不给代价。
- `needed_by`：有时效要求时必须写。
- `blocking_reason`：说明为什么不拍板就不能继续。

## 结果文档如何配合

当任务进入拍板状态时，`result.md` 至少应补入：

- 当前已完成到哪一步。
- 本次待拍板事项编号。
- 主公看完后能直接拍板的建议结论。
- 若未拍板，当前停在哪。

## 推荐汇报顺序

1. 先给一句话结论。
2. 再给主公需要拍的具体问题。
3. 再列选项与影响。
4. 最后给推荐项与理由。

## 收口检查

- 是否已把待拍板事项写入 `gates.yaml`，而不是只留在聊天里。
- 是否已把推荐项和理由写清楚。
- 是否已说明不拍板时的阻塞点。
- 是否已在 `state.yaml` 中切到 `waiting_gate` 或恢复到下一个真实状态。
- 是否已在 `decision-log.md` 中记录为何进入拍板。
- 是否已在 `result.md` 中写明当前完成度与待拍板编号。
- 拍板完成后，是否已把 `gates.yaml` 状态改为 `decided` 或 `dropped`。

## 与维护层总入口的关系

- 需要准备拍板包时，先看 `docs/40-执行/13-维护层总入口.md`。
- 需要半自动生成首个拍板包骨架时，看 `docs/40-执行/16-拍板包半自动模板.md`。
- 进入拍板包细节时，以本文为准。
- 若存在多个 gate 或异常并存，先看 `docs/40-执行/19-多 gate 与多异常并存处理规则.md`。
- 主公完成拍板后，如需半自动回写结果，看 `docs/40-执行/17-拍板结果回写模板.md`。
- 拍板完成后，再回到 `docs/40-执行/14-维护层动作矩阵与收口检查表.md` 做统一收口。

## 本文档的价值

- 把拍板动作从临场发挥变成固定流程。
- 降低主公看到半成品判断的概率。
- 为后续拍板包半自动生成保留稳定结构基础。
