# 试用接待与跟进话术

最后更新：2026-04-02
状态：现行标准件
适用范围：第一轮公开试用后的接待与跟进
说明：目标不是“显得专业”，而是尽快把有兴趣的人接进第一次真实试用

## 一句话结论

有人回复“我想试”之后，不要临时现编。直接用下面的话术，把人尽快带到“拿到仓库 → 跑初始化 → 回到 Codex 面板继续”这条线上。

## 先记住这 3 条

1. 先确认对方是不是 Windows 开发者。
2. 先把第一轮动作压到最少，不一次讲太多。
3. 先把人带进真实试用，再收集卡点。

## 场景 1：有人第一次回复“我想试”

```text
好呀，我这套现在先支持 Windows 开发者试用。
你如果方便，我先把仓库和最短上手路径发你：
1. 先跑初始化
2. 必要时验真一下
3. 再建一个任务，然后回官方 Codex 面板直接继续

你试的时候如果卡住，直接把报错或卡点发我就行，我会按第一轮试用帮你一起走通。
```

## 场景 2：确认对方适不适合试

```text
我先确认两个小问题：
1. 你现在主要是不是在 Windows 上用 Codex？
2. 你平时是不是会在官方 Codex 面板里连续做项目？

如果这两个都差不多符合，这一轮试用就比较适合你。
```

## 场景 3：把最短上手路径发给对方

```text
最短上手路径就两步：

第一步，在仓库根目录跑：
powershell.exe -ExecutionPolicy Bypass -File .\codex-home-export\initialize-workspace.ps1

第二步，建一个你自己的任务：
powershell.exe -ExecutionPolicy Bypass -File .\codex-home-export\new-task.ps1 -Title "你的任务标题"

跑完后，回官方 Codex 面板，新开会话，把脚本打印出来的那段话直接贴进去就能继续。
```

## 场景 4：对方说“我不会配”

```text
没关系，这一轮本来就是想看看第一次上手哪里最重。
你不用自己硬啃，直接把你卡住的那一步发我就行。
我先帮你把第一轮走通，顺手也把你的卡点记下来，后面我会继续把这一步压短。
```

## 场景 5：24 小时没回复，轻跟进一次

```text
我来轻轻追一下进度：
你那边如果已经开始试了，最先卡住的是哪一步？
如果还没开始，也没关系，你只要回我一句“卡在初始化 / 卡在验真 / 卡在建任务 / 还没开始”，我就知道下一步怎么帮你。
```

## 场景 6：72 小时没动静，再问最后一次

```text
我最后补问一次：
如果你还想试，我可以继续帮你把第一轮走通；
如果你暂时没空，也完全没关系，我就先不追着问了。
以后你想再开，我这边还有现成入口和上手说明。
```

## 场景 7：对方已经试了，但卡住了

```text
收到。你先别急着补背景，直接发我这 3 个东西就行：
1. 你卡在哪一步
2. 你看到的原话或截图
3. 你本来以为下一步会发生什么

这样我能更快判断，到底是命令重、说明不清，还是回到 Codex 面板继续这一步不顺。
```

## 场景 8：对方顺利试完，怎么追问最有用

```text
太好了。那我只追 3 个最有用的问题：
1. 你第一次最担心的是哪一步？
2. 真正开始试后，最卡的是哪一步？
3. 如果只允许我下一刀改一件事，你最希望我改什么？
```

## 发给对方之前先检查这 4 项

1. 有没有先确认对方是否适合这一轮试用。
2. 有没有只给最短路径，而不是一次甩太多文档。
3. 有没有把主入口写成官方 Codex 面板。
4. 有没有留下一个明确下一动作。
