# 任务包模板

最后更新：2026-03-29
状态：现行标准件
适用阶段：`V4-Trial`
说明：本模板以当前仓为唯一现行落点；用于新任务的最小起包，不依赖旧仓路径

## 一句话目标

把新任务从“口头描述”变成“可继续、可恢复、可验收”的标准任务包。

## 使用步骤

1. 先确定任务 ID，格式必须为 `v4-trial-<三位序号>-<语义名>`。
2. 在 `.codex/chancellor/tasks/<task-id>/` 下创建 5 个标准文件。
3. 在 `active-task.txt` 中写入当前激活任务 ID。
4. 按推进过程持续更新 `state.yaml`、`decision-log.md`、`gates.yaml`、`result.md`。

## 目录模板

```text
.codex/chancellor/
  active-task.txt
  tasks/
    <task-id>/
      contract.yaml
      state.yaml
      decision-log.md
      gates.yaml
      result.md
```

## `active-task.txt` 模板

```text
v4-trial-002-task-package-template
```

## `contract.yaml` 模板

```yaml
task_id: v4-trial-xxx-your-task
title: 任务标题
goal: >-
  用一句完整的话写清任务要达成什么结果。
constraints:
  - 不得突破的边界 1
  - 不得突破的边界 2
acceptance:
  - 可验收标准 1
  - 可验收标准 2
must_gate:
  - 需要主公拍板的事项；若无则写 []
default_auto:
  - 可默认自动推进的低风险动作
source_refs:
  - 优先写当前目录内文件路径
planning_hint: >-
  当前主假设；若任务低不确定性可写 optional。
```

## `state.yaml` 模板

```yaml
task_id: v4-trial-xxx-your-task
status: drafting
risk_level: low
next_action: 下一步最小动作
blocked_by: []
updated_at: '2026-03-29 00:07:51'
phase_hint: optional_phase
plan_step: 当前最小推进步
verify_signal: 当前步完成后应看到的验证信号
```

## `decision-log.md` 模板

```md
# 决策记录

## 2026-03-29 00:07:51

- 决策：本次关键决定
- 原因：为什么这样做
- 证据：依据了哪些当前目录内文档或结果
- 影响：后续会影响什么
- 路线修正：若验证失败，下一轮改走哪条路径
```

## `gates.yaml` 模板

```yaml
task_id: v4-trial-xxx-your-task
items:
  - gate_id: gate-001
    status: pending
    question: 是否允许执行某项边界变更
    recommendation: 选项 A
    options:
      - name: 选项 A
        impact: 短期成本较高，但长期复利更高
      - name: 选项 B
        impact: 短期更快，但后续维护成本更高
    needed_by: '2026-03-29 12:00:00'
    blocking_reason: 未拍板前不能继续下一步
```

如当前无待拍板事项，显式写：

```yaml
task_id: v4-trial-xxx-your-task
items: []
```

## `result.md` 模板

```md
# 结果摘要

## 已完成

- 已完成事项 1
- 已完成事项 2

## 验证证据

- 文档或文件路径 1
- 文档或文件路径 2

## 遗留事项

- 尚未完成事项 1
- 尚未完成事项 2

## 下一步建议

- 下一步最小动作 1
- 下一步最小动作 2

## 规划复核

- 当前主假设是否成立：成立 / 不成立 / 待补证据
- 当前最小推进步是否完成：是 / 否
- 下一轮是否需要改路：否 / 是（说明原因）
```

## 填写原则

- 目标、边界、验收必须写具体，不写空话。
- `source_refs` 优先引用当前目录内文件，不直接依赖旧仓路径存活。
- `gates.yaml` 无待拍板时也必须显式写 `items: []`。
- 如存在待拍板事项，字段结构与汇报顺序以 `docs/40-执行/15-拍板包准备与收口规范.md` 为准。
- `result.md` 不能只写结论，必须写验证证据、遗留事项与下一步建议。
- 若任务不确定性较高，应补写主假设、最小推进步、验证信号与规划复核。
- 若通过脚手架起包，`result.md` 应保留对收口检查表的引用，避免起包后失联。

## 与公开仓的关系

- 本文档可进入公开仓，作为固定模板说明。
- 真实运行态任务包继续只留本地，不进入公开仓。
- 公开仓中的 `.codex/chancellor/tasks/README.md` 只保留结构说明，不放真实任务内容。
