# 任务包模板

最后更新：2026-04-01
状态：现行标准件
适用阶段：`V4-Trial / V4-Target`
说明：本模板以当前仓为唯一现行落点；用于新任务的最小起包，不依赖旧仓路径

## 一句话目标

把新任务从“口头描述”变成“可继续、可恢复、可验收”的标准任务包。

## 使用步骤

1. 先确定任务 ID，格式必须为 `v4-<trial/target>-<三位序号>-<语义名>`。
2. 在 `.codex/chancellor/tasks/<task-id>/` 下创建 5 个标准文件。
3. 在 `active-task.txt` 中写入当前激活任务 ID。
4. 按推进过程持续更新 `state.yaml`、`decision-log.md`、`gates.yaml`、`result.md`。
5. 若任务需要跨聊天续做，额外刷新 `progress-snapshot.md` 与 `handoff.md`。

## 目录模板

```text
.codex/chancellor/
  active-task.txt
  tasks/
    <task-id>/
      contract.yaml
      state.yaml
      decision-log.md
      gates.yaml
      result.md
      progress-snapshot.md  # 可选，任务级进度快照
      handoff.md            # 可选，跨聊天交班单
      tech-spec.md  # 可选，复杂任务（estimated_hours > 4）需要
```

## `active-task.txt` 模板

```text
v4-target-004-new-task-target-support
```

## `contract.yaml` 模板

```yaml
task_id: v4-target-xxx-your-task
title: 任务标题
goal: >-
  用一句完整的话写清任务要达成什么结果。
  
  【VibeCoding 要求】必须写清楚具体目标，不能模糊。
  错误示例：优化性能
  正确示例：把登录接口响应时间从 2 秒降到 500 毫秒

user_scenario: >-
  【VibeCoding 要求】用户场景必须包含以下 4 要素：
  - 谁：[具体用户角色]
  - 在什么情况下：[具体场景]
  - 要做什么：[具体操作]
  - 期望什么结果：[可验证的结果]

technical_approach: >-
  【VibeCoding 要求】技术方案概要必须包含：
  - 改动哪些文件
  - 核心逻辑是什么
  - 有什么风险点

constraints:
  - 不得突破的边界 1
  - 不得突破的边界 2
acceptance:
  - 【VibeCoding 要求】验收标准必须可验证、可量化，不能写"功能正常"
  - 可验收标准 1
  - 可验收标准 2
must_gate:
  - 需要主公拍板的事项；若无则写 []
default_auto:
  - 可默认自动推进的低风险动作
source_refs:
  - 优先写当前目录内文件路径
planning_hint: >-
  当前主假设；若任务低不确定性可写 optional。
planning_required: false  # 是否需要 tech-spec.md（复杂任务设为 true）
planning_status: not_required  # pending/approved/rejected/not_required
estimated_hours: 0  # 预估工时，> 4 自动触发 planning_required
```

## `state.yaml` 模板

```yaml
task_id: v4-target-xxx-your-task
status: drafting  # drafting/planning/ready/running/waiting_gate/waiting_assist/verifying/done/paused/ready_to_resume
risk_level: low
next_action: 下一步最小动作
blocked_by: []
updated_at: '2026-03-29 00:07:51'
phase_hint: optional_phase
plan_step: 当前最小推进步
verify_signal: 当前步完成后应看到的验证信号
```

**状态机说明**：
- `drafting`：起草阶段，允许编辑任务包文件，禁止修改代码文件
- `planning`：规划阶段，允许编辑 tech-spec.md，禁止修改代码文件
- `ready` / `running`：执行阶段，允许修改代码文件

## `decision-log.md` 模板

```md
# 决策记录

## 2026-03-29 00:07:51

- 决策：本次关键决定
- 原因：为什么这样做
- 证据：依据了哪些当前目录内文档或结果
- 影响：后续会影响什么
- 路线修正：若验证失败，下一轮改走哪条路径
```

## `gates.yaml` 模板

```yaml
task_id: v4-target-xxx-your-task
items:
  - gate_id: gate-001
    status: pending
    question: 是否允许执行某项边界变更
    recommendation: 选项 A
    options:
      - name: 选项 A
        impact: 短期成本较高，但长期复利更高
      - name: 选项 B
        impact: 短期更快，但后续维护成本更高
    needed_by: '2026-03-29 12:00:00'
    blocking_reason: 未拍板前不能继续下一步
```

如当前无待拍板事项，显式写：

```yaml
task_id: v4-target-xxx-your-task
items: []
```

## 填写原则

- 目标、边界、验收必须写具体，不写空话。
- `source_refs` 优先引用当前目录内文件，不直接依赖旧仓路径存活。
- `gates.yaml` 无待拍板时也必须显式写 `items: []`。
- 如存在待拍板事项，字段结构与汇报顺序以 `docs/40-执行/15-拍板包准备与收口规范.md` 为准。
- `result.md` 不能只写结论，必须写验证证据、遗留事项与下一步建议。
- 若任务不确定性较高，应补写主假设、最小推进步、验证信号与规划复核。
- 若通过脚手架起包，`result.md` 应保留对收口检查表的引用，避免起包后失联。
- 若任务要跨聊天续做，`handoff.md` 应优先写清本轮背景、整体目标、建议做法与下一步；`progress-snapshot.md` 应优先写清当前状态、验证信号与已有结果。

## 与公开仓的关系

- 本文档可进入公开仓，作为固定模板说明。
- 真实运行态任务包继续只留本地，不进入公开仓。
- 公开仓中的 `.codex/chancellor/tasks/README.md` 只保留结构说明，不放真实任务内容。
