# 维护层动作矩阵与收口检查表

最后更新：2026-03-29
状态：现行维护标准件
适用范围：当前仓全部维护层动作
说明：本文件用于把维护层常见动作的触发条件、入口文档、产出要求、公开边界与收口动作固定下来，减少临场判断成本。

## 一句话结论

先用动作矩阵判断该走哪条维护路径，再按收口检查表完成留痕、导航同步、提交与推送。

## 维护层动作矩阵

## 默认决策顺序

1. 先判断动作属于“提交、起包、归档、入口同步、拍板准备、治理复核”哪一类。
2. 再进入对应主入口文档，不跨文档来回跳。
3. 当前轮若影响公开口径或现行标准件，收口前追加一次治理审计复核。
4. 动作完成后，统一执行收口检查表。
5. 只有通过收口检查，才算本轮维护层动作结束。

## 收口检查表

### 基础收口

- 是否已经明确本轮动作属于哪一种维护类型。
- 是否已经把结果沉淀到当前目录内的现行文档或归档文档。
- 是否已经生成带时间戳的本地日志，写清动作、结果、理由、下一步。
- 若新增或变更现行标准件，是否已经同步 `README.md`、`docs/README.md`、`docs/00-导航/02-现行标准件总览.md`、`docs/40-执行/README.md`。
- 是否已经按串行流程完成 `commit`、`pull --rebase`、`push`。
- 是否已经给出下一步建议，并说明是否需要主公拍板。

### 治理审计补充检查

- 当前轮关键配置或关键口径，是否已经说明来源、版本或现行依据。
- 当前轮关键决策，是否已经在 `decision-log.md` 或现行文档中写明原因、依据与影响。
- 当前轮关键输出，是否已经能追溯到现行件、验证结果、拍板链路或异常链路。
- `README.md`、`docs/README.md`、`docs/00-导航/02-现行标准件总览.md`、`docs/40-执行/12-V4-Target-实施计划.md` 之间是否存在口径漂移；若有，是否已修平。
- 是否已经复核公开仓边界，确保 `.codex/`、`logs/` 等运行态与留痕文件不进入公开仓。

## 公开仓边界提醒

- 公开仓只放正式文档、现行标准件、必要脚本与安全配置。
- `.codex/chancellor/tasks/`、`.codex/chancellor/active-task.txt`、`logs/` 继续作为本地运行态与留痕区，不进入公开仓。
- 如未来新增维护层脚本，应优先复用当前仓目录，不引入仓外依赖。

## 推荐搭配关系

- 做提交动作时：先看 `10`，做完后按本文收口。
- 做起包动作时：先看 `11`，脚手架会写入收口提示，做完后仍按本文逐项收口。
- 做归档动作时：先看 `01-执行区证据稿归档规则.md`，做完后按本文收口。
- 做入口变更时：先看 `13`，再按本文核对是否同步完全。
- 做拍板包准备时：先看 `15`，拍板完成后再按本文统一收口。
- 做拍板包半自动起手时：先看 `16`，生成后仍回到 `15` 与本文完成闭环。
- 做拍板结果回写时：先看 `17`，回写后按本文完成统一收口。
- 做异常路径或回退记录时：先看 `18`，记录后按本文完成统一收口。
- 做多 gate / 多异常裁决时：先看 `19`，先定主状态，再选单项模板落盘。
- 做复杂并存汇报落盘时：先看 `20`，统一写入 `result.md` 与 `decision-log.md`，必要时同步 `state.yaml`。
- 做公开口径变更、复杂裁决或提交推送前：追加一次 `08-V4-治理审计候选规范.md` 对应的治理审计复核。

## 本文档的价值

- 把维护层“怎么选动作、怎么收口”写成固定套路。
- 降低维护动作对执行者个人经验的依赖。
- 为后续更强自动化保留稳定的人类控制平面。
