# V4 MVP 边界与首批任务包建议

最后更新：2026-03-28 23:24:58
来源：旧参考仓 `F:\260321everything-codex 丞相模式测试项目\docs\chancellor-mode\v4\`
状态：提炼稿，可作为新仓第一轮拍板输入

## 一句话判断

当前真正瓶颈不是“缺更多文档”，而是“缺一份能直接指导开工的 `MVP` 边界冻结稿”。

如果边界不先冻结，后续无论是迁旧稿、补任务包，还是决定是否接入 `codex-home-export/`，都会再次扩散。

## 从旧稿提炼出的最小共识

### 1. `V4` 的真正目标

- 对外只保留一个统一入口，不再让主公先判断该用哪一套版本。
- 用文件驱动控制平面承接长任务，避免任务继续漂在聊天历史里。
- 低风险动作可自动推进，中高风险动作进入治理或拍板。
- 保持当前官方面板主入口，不为了做 `V4` 先重造独立前台。

### 2. 当前轮 `MVP` 必须保留的核心能力

- 有统一入口语义：面板输入 `丞相：...` 后，系统行为稳定一致。
- 有最小任务合同：至少能表达目标、限制、验收、风险和来源。
- 有最小状态文件：至少能表达当前状态、下一步、阻塞、更新时间。
- 有决策记账与待拍板文件：关键判断不再只留在聊天里。
- 有任务包落点：运行态放在 `.codex/chancellor/tasks/`，不再混进 `docs/`。

### 3. 当前轮明确不做

- 不做独立 `Web UI` 或独立 `CLI`。
- 不做多 `Agent` 编排。
- 不做完整长期记忆系统。
- 不做全量治理默认化。
- 不做整包迁入旧仓 `codex-home-export/`。
- 不做把旧仓所有 `V4` 文档整体搬进新仓。

## 新仓建议冻结的 `MVP` 边界

### 必须做

1. 冻结 `V4 MVP` 范围，形成单一口径文档。
2. 固化任务包最小结构，以 `.codex/chancellor/tasks/<task-id>/` 为唯一运行态目标。
3. 明确面板入口与任务包的关系：入口负责触发，任务包负责承载。
4. 固化最小验收口径：任务能创建、能推进、能暂停、能恢复、能留痕。

### 暂不做

1. 不把旧仓的 `v4-state` 机制原样搬来继续用。
2. 不在当前轮决定完整安装链重构。
3. 不在当前轮展开完整目标态总蓝图实施。
4. 不在当前轮扩出测试矩阵、发布链、第二机链路。

## 首批任务包建议

### 建议任务包 1：`v4-trial-001-mvp-boundary-freeze`

**目标**

把新仓本轮 `MVP` 的“做 / 不做 / 验收”一次写死，作为后续任务的总合同。

**最小验收**

- 有一份唯一边界文档。
- 能明确回答“这轮为什么不迁整包 `codex-home-export/`”。
- 能明确回答“哪些能力必须先跑通，哪些延后”。

**为什么先做**

这是当前最小可验证推进点；边界不冻，任何执行动作都会变成边做边改方向。

### 建议任务包 2：`v4-trial-002-task-package-template`

**目标**

把 `.codex/chancellor/tasks/` 下的标准任务包模板定稿，补齐 `contract.yaml`、`state.yaml`、`decision-log.md`、`gates.yaml`、`result.md` 的最小字段与示例。

**最小验收**

- 有可复用模板。
- `active-task.txt` 的使用规则明确。
- 新会话可仅靠任务包理解当前任务。

**为什么排第二**

旧稿已经反复证明：没有任务包，长任务仍会回到靠聊天漂移的老路。

### 建议任务包 3：`v4-trial-003-panel-entry-acceptance`

**目标**

把面板入口行为与人工验收步骤写成固定流程，确保入口稳态和任务包衔接一致。

**最小验收**

- `丞相：` 首句触发风格稳定。
- `丞相状态` 能读出当前任务信息。
- 入口相关改动后，有“重新安装 + 新开会话 + 首句验板”闭环。

**为什么排第三**

入口不稳，前两项即使写好也无法可靠试跑。

## 当前不建议立即立项的事项

- `codex-home-export/` 整体迁入或重写
- 独立前端 / 独立命令行
- 全量版本治理与发布链恢复
- 多任务并发和复杂治理规则

这些事项都应该等前 3 个任务包跑通后，再决定是否进入下一阶段。

## 建议采用的成功标准

- 主公不再需要先判断入口，应直接从面板进入。
- 同一任务能跨轮恢复，不依赖长聊天历史。
- 风险、决策、结果能稳定写入文件，而不只留在回复里。
- 运行态与文档区彻底分离，不再回到 `docs/.../v4-state/` 模式。

## 对旧仓资产的处理建议

- `docs/chancellor-mode/v4/`：继续作为参考源，不搬全套。
- `codex-home-export/`：先只保留“现役安装母体”的参考身份，延后决策。
- `tests/`：仅在新仓 `MVP` 边界与结构稳定后，再挑保护性测试迁入。

## 本稿结论

新仓下一步最合理的推进，不是继续补背景，而是先把“`MVP` 边界冻结”定为第一任务，再围绕任务包模板和入口验收做连续两步推进。

换句话说：先立法，再立任务包，再验入口；不要反过来。
