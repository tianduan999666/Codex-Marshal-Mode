# 丞相模式（Chancellor Mode）

给官方 `Codex` 面板加一层更稳、更清楚、更适合中文用户的任务入口。

## 它解决什么痛点

- 开工前要反复解释背景，效率低。
- 换聊天后上下文容易断，任务接不上。
- 版本、真源、运行态不清楚，容易误判。
- 自检、升级、失败提示太硬，不知道下一步怎么做。
- 改了受管文件后，容易出现“同版本不同内容”。

## 我们做了什么

- 把日常入口统一成 `传令：XXXX`。
- 固定了 `传令：版本`、`传令：状态`、`传令：升级`。
- 做了 `传令：交班`、`传令：接班`，支持换聊天继续干活。
- 加了任务级进度快照，减少重复交代背景。
- 把安装、升级、自检、回滚收敛成固定维护入口。
- 给主链失败场景补了更清楚的人话收口。
- 建了版本纪律：受管文件变更后必须 bump 版本并重验。

## 和 ECC 的联系与区别

| 项目 | 丞相模式 | ECC |
| --- | --- | --- |
| 联系 | 借鉴了“让开发更稳、更顺、更少重复动作”的思路 | 是重要参考来源之一 |
| 重点 | 官方 `Codex` 面板、中文单入口、单机稳态、状态可感知 | 多 Agent、Skills、Hooks、跨 harness 能力包 |
| 目标 | 把 `Codex` 日常开工链做到稳、狠、好用 | 做更大范围的工程化生态 |
| 当前判断 | 不是 ECC 复刻版，也不是 ECC 改名版 | 也不是丞相模式的替代品 |

一句话：丞相模式不是照抄 ECC，而是吸收它有价值的部分，专门服务官方 `Codex` 面板这条主链。

## 真实功能对比表

| 功能 | 丞相模式 | ECC（真实仓） |
| --- | --- | --- |
| 中文单入口 | 有，主入口固定为 `传令：XXXX` | 不是主打；更偏 commands / skills / agents / hooks 组合 |

| 交班 / 接班 | 有；公开命令直接可用 | 未见同等公开单命令口径；更偏通过 sessions / workflows / agents 组合解决 |
| 本机运行态验真 | 有；安装、文件级验真、完整自检是一条链 | 有安装与同步能力，但不是围绕“单机中文面板稳态”设计 |
| Skills 能力库 | 当前不是主打，走少量高价值补件路线 | 强；README 当前写明 `180 skills` |

| Legacy commands | 当前公开命令刻意收紧，只保留少量固定命令 | 强；README 当前写明 `79 legacy command shims` |
| Hooks | 当前弱，主要靠显式脚本和真源路由 | 强；README 明确有 hooks 体系 |
| 跨平台 / 跨 harness | 当前不主打，聚焦官方 `Codex` 面板 | 强；README 明确覆盖 Claude Code、Codex、Cursor、OpenCode、Gemini 等 |
| 插件 / 安装面 | 当前主打本机稳定安装与固定维护入口 | 强；README 同时提供 plugin 安装、脚本安装和多目标安装 |
| 大型自动化运行时 | 当前不主打 | 有；README 明确 `multi-*` 还依赖额外 `ccg-workflow` runtime |
| 产品定位 | 稳定开发控制面 | 大范围工程能力包 |

一句话：

- 丞相模式更像“把 `Codex` 日常开工这条线做稳”。
- ECC 更像“给多种 agent/harness 提供一整套工程能力包”。

## 为什么不叫 EC（everything codex）

- 因为它不是“everything codex”路线。
- `EC` 这个名字会让人误以为它要做 `Codex` 全家桶、全能力平台、全生态封装。
- 但丞相模式当前不追这个方向。
- 它的定位更窄也更实：`传令` 单入口、中文状态感知、真源统一、本机稳态。
- 所以它直接叫 `丞相模式`，比叫 `EC` 更准确，不会把目标带偏成“大而全平台”。

## 丞相模式特点

- 中文单入口：只记 `传令：XXXX`。
- 状态可感知：版本、检查时间、当前任务更容易看懂。
- 真源统一：公开口径尽量从同一套真源渲染。
- 单机稳态：默认不乱改用户现有全局模型配置。
- 维护收口清楚：安装、升级、自检、回滚入口固定。
- 跨聊天连续：`交班 / 接班` 能直接续上任务。

## 用它有什么好处

- 开工更快：直接下任务，不先折腾脚本。
- 接力更稳：聊天切换不容易断线。
- 状态更清：版本、运行态、当前任务更容易看懂。
- 维护更稳：安装、升级、自检、回滚路径固定。
- 风险更低：默认不会覆盖用户现有的 `~/.codex/config.toml`。

## 最短安装

```powershell
git clone https://github.com/tianduan999666/Codex-Chancellor-Mode.git
cd Codex-Chancellor-Mode
.\install.cmd
```

普通用户只做这 3 行。

## 命令怎么用

| 命令 | 你怎么说 | Codex 预期动作 | 预期结果 |
| --- | --- | --- | --- |
| 日常做事 | `传令：修一下登录页` | 接任务，必要时自检，建任务，开始执行 | 直接进入干活 |
| 看状态 | `传令：状态` | 读取当前状态栏 | 看到版本、检查、模式、当前任务 |
| 看版本 | `传令：版本` | 读取版本真源 | 看到版本号、版本来源、真源路径 |
| 升级 | `传令：升级` | 只处理丞相自身升级或同步 | 当前丞相升级，不碰用户项目 |
| 继续当前任务 | `传令：继续` | 优先续上当前激活任务 | 不重复起新任务 |
| 交班 | `传令：交班` | 生成 `progress-snapshot.md` 和 `handoff.md` | 新聊天可直接续上 |
| 接班 | `传令：接班` | 读取当前任务的交班材料 | 快速恢复上下文继续干活 |

## 哪些丢给 Codex，哪些必须人工做

| 类型 | 适合丢给 Codex | 必须人工操作 |
| --- | --- | --- |
| 开发任务 | 页面修改、脚本修改、文档更新、测试补齐、版本 bump | 业务目标拍板、最终验收拍板 |
| 日常推进 | `传令：XXXX`、`传令：继续`、`传令：交班`、`传令：接班` | 判断今天先做哪条主线 |
| 状态核对 | `传令：状态`、`传令：版本`、自检、冒烟、追平状态表 | 判断是否接受当前风险 |
| 安装维护 | 安装、自检、升级、回滚的脚本执行与结果解释 | 提供可用 key、处理账单/额度、真人面板验证 |
| 环境问题 | Codex 可帮你定位和解释 | provider 充值、换 key、权限开通这类外部动作必须人工做 |

一句话：

- 丢给 Codex 的，是“可执行、可验证、可回滚”的动作。
- 必须人工做的，是“拍板、授权、付费、真人确认”。

## 其他必要信息

- 普通用户默认回官方 `Codex` 面板使用，不需要长期在终端里操作。
- 当前公开真源在 `codex-home-export/`，版本真源是 `codex-home-export/VERSION.json`。
- 当前这套更适合“官方 `Codex` 面板 + 中文开发控制面”，不追“ECC 全家桶复刻版”或 “everything codex” 大平台路线。
- 维护说明见 `docs/README.md`。
