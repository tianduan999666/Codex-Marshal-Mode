# 丞相模式（Chancellor Mode）

给官方 `Codex` 面板加一层更稳、更清楚、更适合中文用户的任务入口。

## 极简概要

| 你最需要先知道的 | 内容 |
| --- | --- |
| 这是什么 | 一套围绕官方 `Codex` 面板的中文任务入口、状态口径和本机稳态治理层 |
| 有什么好处 | 少解释、少重复、少断档；版本更清楚，任务更连续，维护更稳 |
| 怎么用 | 在已安装官方 `Codex` 的 Windows 环境里额外执行一次 `install.cmd`，然后回面板直接说：`传令：写一个贪吃蛇小游戏` |

当前仓库当前是 GitHub `Public` 仓，并已补齐 `MIT License`。
这个仓本身当前可以免费获取、免费安装、免费使用；但官方 `Codex` 自身的账号、订阅、额度或 API 费用，仍按官方规则另算。
如果你已经在用官方 `Codex`，额外装上这层即可使用。

如果只看 30 秒，就记住这三点：

1. 丞相模式不是另一个聊天产品，而是给官方 `Codex` 补一层更稳的“开工控制面”。
2. 它的目标不是让用户做更多事，而是让用户参与更少，但结果依然靠谱、可追踪、可维护。
3. 日常只需要记住一个入口：`传令：XXXX`。

## 为什么叫“丞相”

这个名字不是为了古风包装，而是为了表达一种工作标准：

- 希望它像诸葛亮受托孤那样，**接手后靠谱**。
- 希望用户只需要给出必要目标，不必每轮都重复铺背景、盯流程、补上下文。
- 希望它不是“看起来很能干”，而是**真的能把事情稳稳接住、交代清楚、持续推进**。

所以它不叫 `EC` 或 `everything codex`。

- `everything codex` 会让人联想到“大而全平台”。
- 丞相模式不是要做 `Codex` 全家桶。
- 它更像一个稳态控制层：让开工、续做、交班、验真、维护都更可靠。

一句话：`丞相` 这个名字强调的是“托付感”和“可靠接手”，不是噱头。

## 它解决什么问题

- 开工前总要反复解释背景，效率低。
- 换聊天后上下文容易断，任务接不上。
- 版本、真源、运行态不清楚，容易误判当前状态。
- 安装、自检、升级、回滚太散，失败后不知道下一步该干什么。
- 改了受管文件后，容易出现“同版本不同内容”。

## 丞相特色

| 特色 | 具体表现 |
| --- | --- |
| 中文单入口 | 日常只记 `传令：XXXX` |
| 状态可感知 | `传令：状态` 固定汇报版本、检查、模式、当前任务等关键信息 |
| 版本可信 | 真源、运行态镜像、受管文件同步和验真是一条链 |
| 跨聊天连续 | `传令：交班`、`传令：接班` 直接衔接上下文 |
| 本机稳态优先 | 默认不乱改用户已有 `~/.codex/config.toml` 的 provider / model / auth |
| 维护入口收口 | 安装、升级、自检、回滚入口固定，不靠临场记忆 |
| 中文结果更像人话 | 失败提示、状态反馈、下一步动作尽量直说，不让用户自己翻脚本猜 |

## 最短开始

```powershell
git clone https://github.com/tianduan999666/Codex-Chancellor-Mode.git
cd Codex-Chancellor-Mode
.\install.cmd
```

当前最稳的适用前提：

- 你已经在 Windows 上装好并能正常使用官方 `Codex`
- 你希望继续用官方 `Codex`，只是额外补一层更稳的入口和维护能力

装完以后，普通用户默认回官方 `Codex` 面板使用，不需要长期在终端里操作。

第一次可以直接试这三句：

- `传令：版本`
- `传令：状态`
- `传令：写一个贪吃蛇小游戏`

## 命令怎么用

| 命令 | 你怎么说 | 丞相会做什么 | 预期结果 |
| --- | --- | --- | --- |
| 日常做事 | `传令：写一个贪吃蛇小游戏` | 接任务，必要时做自检，记录任务，开始执行 | 直接进入干活 |
| 看状态 | `传令：状态` | 读取固定状态栏 | 看到版本、检查时间、模式、当前任务 |
| 看版本 | `传令：版本` | 读取版本真源 | 看到版本号、版本来源、真源路径 |
| 升级 | `传令：升级` | 只处理丞相自身升级或同步 | 当前丞相升级，不碰你的项目 |
| 继续当前任务 | `传令：继续` | 优先续上当前激活任务 | 不重复起新任务 |
| 交班 | `传令：交班` | 生成当前任务快照和交班材料 | 新聊天可直接接上 |
| 接班 | `传令：接班` | 读取交班材料，恢复当前任务上下文 | 快速继续上一轮工作 |

## 对原生 Codex 的影响边界

- 你输入 `传令：XXXX` 时，进入丞相主链。
- 你不输入 `传令：` 时，仍然可以像平时一样继续提问、继续开发、继续改代码。
- 默认安装不会静默覆盖你现有 `~/.codex/config.toml` 里的 `provider / model / auth`。
- 当前仓会把丞相的 `AGENTS.md` 与维护脚本同步到 `~/.codex`，所以它不是“完全零侵入”的空安装。
- 更准确的说法是：**不会把你的 Codex 用坏，也不会强行接管 provider / model / auth；但这台机器会开始加载丞相的工作口径，所以它不是“完全没装过”的原始状态。**

## 哪些适合交给 Codex，哪些必须人工做

| 类型 | 适合交给 Codex | 必须人工做 |
| --- | --- | --- |
| 开发执行 | 页面修改、脚本修改、文档更新、测试补齐、版本 bump | 业务目标拍板、最终验收拍板 |
| 日常推进 | `传令：XXXX`、`传令：继续`、`传令：交班`、`传令：接班` | 决定这轮优先做哪条主线 |
| 状态核对 | `传令：状态`、`传令：版本`、自检、冒烟、运行态复核 | 判断是否接受当前风险 |
| 安装维护 | 安装、自检、升级、回滚脚本执行与结果解释 | 提供可用 key、处理额度/账单、真人确认外部平台状态 |
| 外部条件 | 本地问题定位、流程解释、修复建议 | 付费、授权、权限开通、平台账号操作 |

原则很简单：

- 交给 Codex 的，是“可执行、可验证、可回滚”的动作。
- 需要人工做的，是“拍板、授权、付费、真人确认”。

## 和 ECC 的联系与区别

丞相模式吸收了 ECC 里“让开发更稳、更少重复动作”的优点，但它不是 ECC 复刻版，也不是 ECC 改名版。

ECC 参考链接：

- `Everything Claude Code (ECC)`：<https://github.com/affaan-m/everything-claude-code>

### 真实差异

| 项目 | 丞相模式 | ECC |
| --- | --- | --- |
| 主战场 | 官方 `Codex` 面板 | 更大的 agent / skills / hooks / workflows 能力包 |
| 主目标 | 把中文日常开工链做稳 | 提供更广泛的工程能力生态 |
| 入口风格 | `传令：XXXX` 单入口 | commands / skills / agents / hooks 组合 |
| 状态感知 | 强调版本、检查、模式、当前任务这些固定口径 | 不以固定中文状态栏为主 |
| 交班接班 | 公开命令直接可用 | 更偏 session / workflow 组合 |
| 本机稳态 | 围绕单机运行态一致性、自检、验真设计 | 更偏多工具、多环境工程化能力 |
| 当前取向 | 少而硬、贴近日常开发主链 | 广而全、偏平台化 |

一句话：

- 丞相模式更像“把 `Codex` 日常开工这条线做稳”。
- ECC 更像“给多种 agent / harness 提供一整套工程能力包”。
